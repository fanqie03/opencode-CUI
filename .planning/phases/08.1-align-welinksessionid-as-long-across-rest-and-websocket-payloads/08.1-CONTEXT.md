# Phase 08.1: Align welinkSessionId as Long across REST and WebSocket payloads - Context

**Gathered:** 2026-03-12  
**Status:** Ready for planning  
**Source:** roadmap insertion + protocol gap tracking + codebase scan

<domain>
## Phase Boundary

本 phase 只处理 `welinkSessionId` 的协议类型收敛，目标是让项目内对外暴露或跨服务透传的 `welinkSessionId` 统一为数字型：

- Layer1 REST 响应中的 `welinkSessionId` 必须输出为 `Long`
- Layer1 WebSocket 消息中的 `welinkSessionId` 必须输出为数字节点
- Layer2 Skill Server / Gateway / PC Agent 之间仍会透传 `welinkSessionId` 的 JSON 必须统一为数字型
- 前端与测试不能再依赖 `welinkSessionId: string` 或 `string | number` 的混合口径

本 phase 不扩展到无关协议治理，不重做整体 session 命名体系，也不引入新的兼容字段。
</domain>

<decisions>
## Implementation Decisions

### Locked Decisions

- `welinkSessionId` 的标准类型固定为后端 `Long`、前端 `number`
- `toolSessionId` 继续保持 `String`
- `userId` 继续保持 `String`
- 以“全项目统一”作为目标，而不是仅限 Layer1 对外返回
- 旧的字符串型 `welinkSessionId` 只允许作为临时输入兼容存在于少量解析逻辑中；对外输出必须全部收口为数字
- 本 phase 的完成标志不是“Java 字段改成 Long”，而是“REST / WebSocket / 中转 JSON / 前端类型 / 测试断言”全部一致

### Claude's Discretion

- 允许把整改拆成多波次执行，但必须先稳定 Skill Server 的 Layer1 对外口径
- 是否保留个别上行解析端对字符串数字的容错，可在不破坏输出一致性的前提下按最小改动处理
- 文档收口只覆盖与本 phase 直接相关的协议与 gap tracking 文件，不顺手做无关协议清理
</decisions>
