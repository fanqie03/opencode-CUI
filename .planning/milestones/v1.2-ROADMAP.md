# Milestone v1.2: Gateway Multi-Service Isolation

**Status:** SHIPPED 2026-03-12
**Phases:** 7-9
**Total Plans:** 12

## Overview

v1.2 聚焦让 `gateway` 在同时对接 `skill-server` 与其他上游服务时，既能稳定隔离来源服务和回流响应，又能通过统一 Snowflake ID 基础设施避免跨服务 session / entity 标识冲突。

## Phases

### Phase 7: 多服务来源识别与回流隔离

**Goal**: 将来源服务类型提升为 `gateway` 链路中的显式维度，避免不同上游服务之间发生请求归属混淆和响应错投。  
**Depends on**: Phase 6  
**Plans**: 3 plans

Plans:

- [x] 07-01: 标准 upstream `source` 协议与可信握手绑定
- [x] 07-02: 多 `source` owner 注册与回流隔离
- [x] 07-03: 结构化路由观测与回归覆盖

**Details:**

- `GatewayMessage` 顶层承载 `source` 与 `traceId`
- `gateway` 在握手阶段绑定可信 `source`，并在消息阶段校验一致性
- owner key 升级为 `source:instanceId`
- 回流先按 `source` 分域，再在域内选择 owner
- 同域 fallback 保留，跨域 fallback 禁止
- 关键路由链路补齐结构化日志字段

### Phase 8: 统一 Snowflake ID 基础设施治理

**Goal**: 在 `skill-server + ai-gateway` 中统一 Snowflake ID 基础设施，消除 DB 自增主键依赖，并完成双模块自动化回归验证。  
**Depends on**: Phase 7  
**Plans**: 4 plans

Plans:

- [x] 08-01: 统一 Snowflake bit layout、epoch 和 serviceCode
- [x] 08-02: `skill-server` 主键写入链路切换到应用侧 Snowflake
- [x] 08-03: `ai-gateway` 主键写入链路切换到应用侧 Snowflake
- [x] 08-04: 补齐治理文档、需求同步与阶段验证

**Details:**

- `skill-server` 与 `ai-gateway` 各自引入统一的 Snowflake ID generator
- `skill_session`、`skill_message`、`skill_message_part`、`agent_connection`、`ak_sk_credential` 切换为应用侧生成 `Long` 主键
- 移除 MyBatis 对 `useGeneratedKeys` 的依赖
- 治理文档与阶段验证工件完成回填

### Phase 08.1: Align welinkSessionId as Long across REST and WebSocket payloads

**Goal**: 收敛协议实现中残留的字符串型 `welinkSessionId`，确保 REST、WebSocket 与相关中转消息对外统一输出数字型会话 ID。  
**Depends on**: Phase 8  
**Plans**: 3 plans

Plans:

- [x] 08.1-01: `skill-server` Layer1 对外协议与回流解析收口
- [x] 08.1-02: Layer2 `ai-gateway` / `pc-agent` 类型与报文对齐
- [x] 08.1-03: `skill-miniapp` 收口与文档回补

**Details:**

- `skill-server` 不再对外输出字符串型 `welinkSessionId`
- `ai-gateway`、`pc-agent`、`skill-miniapp` 统一到数字型会话 ID
- 协议文档示例同步为数字型 `welinkSessionId`

### Phase 9: Formalize Phase 7 verification and traceability

**Goal**: 为 Phase 7 的多服务来源隔离能力补齐正式 verification artifact、requirement 证据链和必要的 traceability 同步，使 v1.2 milestone audit 不再因验证工件缺失而阻塞。  
**Depends on**: Phase 08.1  
**Plans**: 2 plans

Plans:

- [x] 09-01: 回填 Phase 7 正式 verification 与 summary 工件
- [x] 09-02: 同步 traceability，并为 Phase 9 自身补齐 verification / summary

**Details:**

- 新增 `07-VERIFICATION.md`
- 新增 `07-01/02/03-SUMMARY.md`
- 补齐 `ROUT-01`、`ROUT-02`、`SAFE-01` 的正式证据链
- 更新顶层 planning 状态，支持 milestone audit 重新通过

## Milestone Summary

**Decimal Phases:**

- Phase 08.1: Align welinkSessionId as Long across REST and WebSocket payloads

**Key Decisions:**

- 将“来源服务”提升为 `gateway` 的显式路由维度
- 采用统一 Snowflake ID 方案，而不是继续依赖数据库自增主键
- 将 `welinkSessionId` 的对外协议基线统一收口到数字型
- 通过单独 gap closure phase 回填正式验证工件，而不是放宽审计标准

**Issues Resolved:**

- 解决多上游服务接入时的来源归属与回流错投风险
- 解决跨服务 session / entity 标识冲突风险
- 解决 `welinkSessionId` 在 REST / WebSocket / relay / frontend 中的类型漂移
- 解决 Phase 7 缺少正式 verification artifact 导致 milestone audit 阻塞的问题

**Issues Deferred:**

- 尚未执行真实测试数据库清库重建与跨服务并发 UAT
- 协议文档仍有历史编码噪声
- Phase 8 / 08.1 的既有 summary frontmatter 仍不完整

**Technical Debt Incurred:**

- milestone audit 对 Phase 8 / 08.1 仍包含少量人工核对
- 联调验证和文档编码治理被接受为后续工作
