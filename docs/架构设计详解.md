# OpenCode-CUI 架构设计详解

## 1. 架构概览

### 1.1 系统定位

OpenCode-CUI 是一个**分布式事件驱动架构**系统，采用**透明中继模式**连接多个独立的 IDE 工具（如 OpenCode）与云端 AI 服务。系统通过 WebSocket 长连接和 Redis Pub/Sub 实现跨实例的消息协调，支持多用户、多设备、多会话的并发管理。

### 1.2 核心特点

- **协议透明**：Gateway 不解析 OpenCode 事件，直接透传给 Skill Server，保持协议独立性
- **多实例可扩展**：通过 Redis Pub/Sub 实现无状态的水平扩展
- **客户端适配**：支持多种 IDE 工具接入，通过统一的 WebSocket 协议和消息信封
- **会话持久化**：所有消息存储在数据库，支持离线恢复和历史查询
- **实时推流**：Skill miniapp 通过 WebSocket 订阅会话流，实时接收 AI 响应

## 2. 组件详解

### 2.1 AI-Gateway (端口 8081)

**职责**：连接管理、AK/SK 认证、事件透传、多实例协调

#### 2.1.1 关键类

**AgentWebSocketHandler**
- 处理 PCAgent WebSocket 连接的生命周期
- 在握手阶段验证 AK/SK 签名
- 处理消息类型：`register`, `heartbeat`, `tool_event`, `tool_done`, `tool_error`, `session_created`
- 维护 `sessionId -> agentId` 映射用于断开连接时清理

**AkSkAuthService**
- HMAC-SHA256 签名验证
- 时间戳窗口验证（±5 分钟）
- Nonce 重放检测（Redis SET NX，TTL 5 分钟）
- 常数时间比较防止时序攻击

**EventRelayService**
- 维护 `agentId -> WebSocketSession` 映射（本地连接）
- 订阅 Redis `agent:{agentId}` 通道接收跨实例消息
- 发布消息到 Skill Server（通过 SkillServerWSClient）
- 处理 agent 上线/离线事件

**RedisMessageBroker**
- 发布/订阅 Redis 通道
- 支持 `agent:{agentId}` 和 `session:{sessionId}` 两种通道
- 为会话消息添加序列号用于乱序检测

#### 2.1.2 数据库表

**agent_connection**
```sql
CREATE TABLE agent_connection (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  ak_id VARCHAR(255) NOT NULL,
  device_name VARCHAR(255),
  os VARCHAR(50),
  tool_type VARCHAR(50),
  tool_version VARCHAR(50),
  status VARCHAR(20),
  last_seen_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

#### 2.1.3 配置参数 (application.yml)

```yaml
server:
  port: 8081

gateway:
  skill-server:
    ws-url: ws://localhost:8082/ws/internal/gateway
    internal-token: changeme
    reconnect-initial-delay-ms: 1000
    reconnect-max-delay-ms: 30000

  agent:
    heartbeat-timeout-seconds: 90
    heartbeat-check-interval-seconds: 30

  auth:
    timestamp-tolerance-seconds: 300
    nonce-ttl-seconds: 300
```

### 2.2 Skill-Server (端口 8082)

**职责**：会话持久化、消息存储、客户端推流、IM 集成

#### 2.2.1 关键类

**GatewayRelayService**
- 接收来自 Gateway 的消息（通过 Redis Pub/Sub）
- 订阅 `session:{sessionId}` 通道
- 验证序列号检测乱序和丢包
- 路由消息到 SkillStreamHandler（推流）和 SkillMessageService（持久化）
- 发送 invoke 命令回 Gateway（通过 Redis `agent:{agentId}` 通道）

**SkillStreamHandler**
- WebSocket 端点：`/ws/skill/stream/{sessionId}`
- 维护 `sessionId -> Set<WebSocketSession>` 映射（多客户端订阅）
- 推送消息类型：`delta`, `done`, `error`, `agent_offline`, `agent_online`
- 为每条消息添加序列号用于客户端排序

**SkillSessionService**
- 创建/查询/关闭会话
- 更新 `last_active_at` 时间戳
- 定时清理空闲会话（可配置超时时间，默认 30 分钟）
- 关联 agent 和 skill definition

**SkillMessageService**
- 保存用户消息、助手消息、系统消息
- 支持分页查询
- 消息类型：USER, ASSISTANT, SYSTEM

**RedisMessageBroker**
- 与 Gateway 中的 RedisMessageBroker 功能相同
- 发布到 `agent:{agentId}` 通道用于跨实例路由

#### 2.2.2 数据库表

**skill_session**
```sql
CREATE TABLE skill_session (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,

  status VARCHAR(20),

  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

**skill_message**
```sql
CREATE TABLE skill_message (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id BIGINT NOT NULL,
  message_type VARCHAR(20),
  content LONGTEXT,
  metadata JSON,

```

**skill_definition**
```sql
CREATE TABLE skill_definition (

  name VARCHAR(255) NOT NULL,
  description TEXT,
  version VARCHAR(50),
  status VARCHAR(20),
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 3. 通信协议

### 3.1 WebSocket 消息格式

#### 3.1.1 PCAgent -> Gateway

**Register**
```json
{
  "type": "register",
  "deviceName": "MacBook Pro",
  "os": "macOS",
  "toolType": "OPENCODE",
  "toolVersion": "1.2.3"
}
```

**Heartbeat**
```json
{
  "type": "heartbeat"
}
```

**Tool Event (透传)**
```json
{
  "type": "tool_event",
  "sessionId": "123",
  "event": {
    "type": "content_block_delta",
    "delta": { "type": "text_delta", "text": "Hello" }
  }
}
```

**Tool Done**
```json
{
  "type": "tool_done",
  "sessionId": "123",
  "usage": {
    "input_tokens": 100,
    "output_tokens": 50
  }
}
```

#### 3.1.2 Gateway -> PCAgent

**Invoke**
```json
{
  "type": "invoke",
  "agentId": "456",
  "sessionId": "123",
  "action": "chat",
  "payload": {
    "messages": [...]
  }
}
```

#### 3.1.3 Skill Miniapp <- Skill Server

**Stream Message**
```json
{
  "type": "delta",
  "seq": 1,
  "content": {
    "type": "content_block_delta",
    "delta": { "type": "text_delta", "text": "Hello" }
  }
}
```

### 3.2 Redis Pub/Sub 通道设计

| 通道 | 发布者 | 订阅者 | 用途 |
|------|--------|--------|------|
| `agent:{agentId}` | Skill Server | Gateway | 跨实例路由 invoke 命令 |
| `session:{sessionId}` | Gateway | Skill Server | 跨实例路由会话消息 |

### 3.3 消息信封（Envelope）协议

用于统一的元数据管理和版本控制：

```json
{
  "envelope": {
  "version": "1.0.0",
    "messageId": "uuid-v4",
    "timestamp": "2025-03-04T10:30:00Z",
    "source": "OPENCODE",
    "agentId": "456",
    "sessionId": "123",
    "sequenceNumber": 1,
    "sequenceScope": "session"
  },
  "type": "tool_event",
  "payload": { ... }
}
```

### 3.4 序列号与乱序检测

- **序列号范围**：每个会话独立计数，从 1 开始
- **检测策略**：
  - 小间隙（1-5）：记录警告，继续处理
  - 中间隙（6-20）：请求恢复
  - 大间隙（>20）：触发重连
- **实现**：SequenceTracker 类维护每个会话的期望序列号

## 4. 数据流分析

### 4.1 上行流：用户消息 → OpenCode

```
用户在 Skill Miniapp 输入消息
    ↓
useSkillStream.sendMessage()
    ↓
POST /api/skill/session/{sessionId}/message
    ↓
SkillSessionController.sendMessage()
    ↓
GatewayRelayService.sendInvokeToGateway()
    ↓
Redis publish agent:{agentId}
    ↓
Gateway EventRelayService 订阅接收
    ↓
SkillServerWSClient 发送到 PCAgent
    ↓
PCAgent 调用 OpenCode API
    ↓
OpenCode 执行用户请求
```

### 4.2 下行流：OpenCode 响应 → 用户

```
OpenCode 生成事件（content_block_delta, tool_use, etc.）
    ↓
PCAgent 监听事件
    ↓
WebSocket 发送 tool_event 到 Gateway
    ↓
AgentWebSocketHandler 接收
    ↓
EventRelayService.relayToSkillServer()
    ↓
SkillServerWSClient 转发到 Skill Server
    ↓
GatewayRelayService.handleToolEvent()
    ↓
SkillMessageService.saveAssistantMessage() (持久化)
    ↓
SkillStreamHandler.pushToSession() (推流)
    ↓
Redis publish session:{sessionId}
    ↓
Skill Miniapp WebSocket 接收
    ↓
useSkillStream 更新消息列表
    ↓
ConversationView 渲染
```

### 4.3 跨实例路由机制

**场景**：多个 Gateway 实例，PCAgent 连接到实例 A，Skill Server 实例 B 需要发送消息

```
Skill Server B 需要发送 invoke 到 agent:456
    ↓
GatewayRelayService.sendInvokeToGateway()
    ↓
Redis publish agent:456 (所有 Gateway 实例订阅)
    ↓
Gateway A 的 EventRelayService 接收
    ↓
检查本地是否有 agentId=456 的连接
    ↓
有：直接发送到 WebSocket
    ↓
无：消息丢弃（该实例无此 agent）
```

## 5. 核心设计模式

### 5.1 透明中继模式

**定义**：Gateway 不解析或修改消息内容，只负责转发和路由

**优势**：
- 协议独立性：支持多种 IDE 工具，无需修改 Gateway
- 版本兼容性：新的事件类型自动支持
- 低耦合：Gateway 与 OpenCode 解耦

**实现**：
```java
// EventRelayService.relayToSkillServer()
GatewayMessage forwarded = message.withAgentId(agentId);
skillServerRelay.sendToSkillServer(forwarded);
// 消息内容完全不变，只添加 agentId 元数据
```

### 5.2 Redis Pub/Sub 协调

**定义**：使用 Redis 作为消息总线，实现无状态的多实例协调

**优势**：
- 无状态：Gateway/Skill Server 实例可随意增删
- 自动发现：新实例自动订阅相关通道
- 高可用：Redis 故障时消息丢失（可接受，因为有 WebSocket 直连）

**通道设计**：
- `agent:{agentId}`：一对多（多个 Skill Server 实例可发送，一个 Gateway 实例接收）
- `session:{sessionId}`：一对多（一个 Gateway 实例发送，多个 Skill Server 实例接收）

### 5.3 客户端协议适配

**定义**：通过消息信封和工厂方法支持多种客户端

**实现**：
```java
// GatewayMessage 提供静态工厂方法
GatewayMessage.register(deviceName, os, toolType, toolVersion);
GatewayMessage.toolEvent(sessionId, event);
GatewayMessage.invoke(agentId, sessionId, action, payload);

// 支持信封和非信封格式
if (message.hasEnvelope()) {
  // 新格式处理
} else {
  // 兼容旧格式
}
```

### 5.4 AK/SK 认证与防重放

**流程**：
1. 客户端生成时间戳、随机 nonce
2. 计算签名：`HMAC-SHA256(SK, "{AK}\n{timestamp}\n{nonce}")`
3. 发送 WebSocket 握手参数：`ak`, `ts`, `nonce`, `sign`
4. Gateway 验证：
   - 时间戳在 ±5 分钟内
   - Nonce 未被使用过（Redis SET NX）
   - 签名正确（常数时间比较）

**防护**：
- 时间戳窗口：防止重放攻击
- Nonce 一次性：Redis 中 TTL 5 分钟
- 常数时间比较：防止时序攻击

## 6. 扩展性设计

### 6.1 水平扩展策略

**Gateway 扩展**：
- 无状态设计：每个实例独立处理 PCAgent 连接
- Redis 协调：通过 Pub/Sub 实现跨实例消息路由
- 负载均衡：PCAgent 可连接任意 Gateway 实例
**Skill Server 扩展**：
- 无状态设计：会话数据存储在数据库
- Redis 协调：通过 Pub/Sub 接收来自任意 Gateway 实例的消息
- 会话亲和性：同一会话的消息可由不同实例处理

**数据库扩展**：
- 分库分表：按 user_id 或 session_id 分片
- 读写分离：查询操作可使用从库
- 消息归档：定期归档旧消息到冷存储

### 6.2 多 AI 工具支持（未来）

**扩展点**：
- 新增 `tool_type` 字段区分不同工具
- 每个工具有独立的 PC Agent Plugin
- Gateway 透传所有工具的事件
- Skill Server 根据 `tool_type` 适配渲染

**示例**：
```json
{
  "type": "register",
  "toolType": "VSCODE",  // 或 "JETBRAINS", "CURSOR"
  "toolVersion": "1.0.0"
}
```

### 6.3 协议版本兼容

**信封版本**：
```json
{
  "envelope": {
    "version": "1.0.0",  // 协议版本
    ...
  }
}
```

**兼容策略**：
- 新版本 Gateway 支持旧版本客户端（向后兼容）
- 旧版本 Gateway 拒绝新版本客户端（向前不兼容）
- 通过版本号在消息处理中进行适配

## 7. 技术决策分析

### 7.1 为什么选择透传而非服务端解析

**对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 透传 | 协议独立、版本兼容、低耦合 | 无法在 Gateway 层做统一处理 |
| 解析 | 统一处理、易于监控 | 协议绑定、版本维护复杂 |

**选择理由**：
- OpenCode 事件格式复杂且可能变化
- 多工具支持需要协议独立性
- Gateway 只需负责路由，不需理解业务逻辑

### 7.2 为什么需要两个独立服务

**对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单服务 | 简单、部署少 | 职责混乱、扩展困难 |
| 双服务 | 职责清晰、独立扩展 | 部署复杂、通信开销 |

**选择理由**：
- Gateway：连接管理、认证、路由（I/O 密集）
- Skill Server：会话管理、消息存储、推流（计算密集）
- 分离后可独立扩展，Gateway 可处理更多连接

### 7.3 为什么使用 WebSocket 长连接

**对比**：

| 方案 | 优点 | 缺点 |
|----|------|
| WebSocket | 低延迟、双向通信、连接复用 | 连接管理复杂 |
| HTTP Polling | 简单、无状态 | 高延迟、高开销 |
| gRPC | 高性能、类型安全 | 客户端支持差 |

**选择理由**：
- 实时推流需要低延迟
- IDE 工具原生支持 WebSocket
- 连接复用减少开销

### 7.4 为什么使用 Redis Pub/Sub

**对比**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| Redis Pub/Sub | 简单、低延迟、自动发现 | 消息不持久化 |
| RabbitMQ | 持久化、可靠性高 | 复杂、开销大 |
| Kafka | 高吞吐、持久化 | 复杂、延迟高 |

**选择理由**：
- 消息不需持久化（有 WebSocket 直连和数据库备份）
- 低延迟要求
- 简化部署和运维

## 10. 总结

OpenCode-CUI 通过分布式事件驱动架构实现了高效的 IDE-AI 集成。核心设计包括：

1. **透明中继**：Gateway 不解析消息，保持协议独立
2. **Redis 协调**：实现无状态的多实例扩展
3. **会话持久化**：所有消息存储，支持离线恢复
4. **实时推流**：WebSocket 长连接提供低延迟体验
5. **安全认证**：AK/SK 签名 + Nonce 防重放

该架构支持水平扩展、多工具接入、版本兼容，是一个生产级的分布式系统设计。
