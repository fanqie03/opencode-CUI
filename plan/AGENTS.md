# 项目规划文档

## 目的

存放项目规划、需求、架构设计等文档，供团队成员和 AI Agent 参考。本目录是项目的单一真实来源（Single Source of Truth），所有实现决策和验收标准均源于此。

## 关键文件

| 文件 | 描述 |
|------|------|
| `00-方案设计主文档.md` | 整体方案设计，系统边界，架构概览，核心约束 |
| `01-MVP需求文档.md` | MVP 需求，用户故事，验收标准，功能范围 |
| `02-架构设计文档.md` | 详细架构设计，组件职责，通信协议，数据流 |
| `03-项目实施计划.md` | 实施阶段，里程碑，任务分解，依赖关系 |
| `04-测试用例文档.md` | 测试场景，测试用例，验证方法，覆盖范围 |
| `05-端到端集成测试系统实施方案.md` | 集成测试框架，测试工具，自动化策略 |

## AI Agent 工作指南

### 在此目录工作时

**修改需求文档：**
- 保持原有结构和格式（Markdown 表格、代码块、列表）
- 添加新需求时更新验收标准部分
- 更新需求时同步修改相关的架构设计文档
- 所有变更需要在文档中记录变更原因和日期

**架构变更：**
- 架构变更需同步更新 `02-架构设计文档.md`
- 更新组件职责、通信协议、数据流图
- 检查是否影响其他文档（需求、测试用例）
- 在 `00-方案设计主文档.md` 中记录核心约束变更

**添加新需求：**
- 在 `01-MVP需求文档.md` 中添加用户故事
- 定义清晰的验收标准（SMART 原则）
- 更新 `03-项目实施计划.md` 中的任务分解
- 添加对应的测试用例到 `04-测试用例文档.md`

### 文档维护规范

**格式要求：**
- 所有文档使用 Markdown 格式
- 表格用于结构化信息（功能列表、配置参数、API 端点）
- 代码块用于示例和配置（JSON、SQL、YAML、代码片段）
- 列表用于步骤、要点、检查清单
- 标题层级：# 主标题，## 二级标题，### 三级标题

**交叉引用：**
- 需求文档中的功能应对应架构文档中的组件
- 测试用例应覆盖需求文档中的所有验收标准
- 实施计划中的任务应对应需求和架构文档
- 使用相对路径引用：`[需求文档](./01-MVP需求文档.md)`

**版本控制：**
- 每个文档顶部包含"最后更新"时间戳
- 重大变更在文档中添加变更日志
- 保持 Git 提交信息清晰（关联文档变更）

### 常见任务

**添加新功能：**
1. 在 `01-MVP需求文档.md` 中添加用户故事和验收标准
2. 在 `02-架构设计文档.md` 中描述实现方案
3. 在 `03-项目实施计划.md` 中分解任务
4. 在 `04-测试用例文档.md` 中添加测试场景

**修改现有功能：**
1. 更新 `01-MVP需求文档.md` 中的需求描述
2. 检查 `02-架构设计文档.md` 是否需要调整
3. 更新 `04-测试用例文档.md` 中的测试用例
4. 在 `03-项目实施计划.md` 中更新任务状态

**架构优化：**
1. 在 `02-架构设计文档.md` 中提出优化方案
2. 评估对需求的影响（`01-MVP需求文档.md`）
3. 更新测试策略（`04-测试用例文档.md`）
4. 更新实施计划（`03-项目实施计划.md`）

```
    ↓

    ↓

    ↓
03-项目实施计划.md (任务分解)
    ↓
04-测试用例文档.md (验证方案)
    ↓
05-端到端集成测试系统实施方案.md (测试框架)
```

**关键约束：**
- 代码实现应严格遵循 `02-架构设计文档.md` 中的设计
- 所有测试用例应覆盖 `01-MVP需求文档.md` 中的验收标准
- 实施顺序应遵循 `03-项目实施计划.md` 中的里程碑
- 集成测试应使用 `05-端到端集成测试系统实施方案.md` 中的框架

## 文档质量检查清单

在提交文档变更前，请检查：

- [ ] 所有标题层级正确（# 主标题，## 二级标题）
- [ ] 表格格式正确，列对齐
- [ ] 代码块有语言标识（```json, ```sql, ```yaml）
- [ ] 交叉引用使用相对路径
- [ ] 验收标准使用 SMART 原则（具体、可测量、可达成、相关、有时限）
- [ ] 没有过时的信息或冗余内容
- [ ] 文档顶部有"最后更新"时间戳
- [ ] 重大变更有变更日志记录
