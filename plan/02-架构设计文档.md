## 1. 系统概览

### 1.1 设计目标

在现有 IM 聊天工具中通过 "/" 命令框架触发 Skill 交互，使开发者无需离开 IM 即可与本地 OpenCode 进行多轮 AI 编码对话。系统采用**全链路透传**架构，协议适配集中在 Skill 小程序端。

### 1.2 设计原则

| 序号 | 原则 | 说明 |
|---|---|---|
| 1 | Skill 交互嵌入 IM | 通过已有 "/" 命令框架触发，Skill 小程序固定在 IM 聊天窗上方 |
| 2 | PCAgent 使用原生 SDK | 直接使用 `@opencode-ai/sdk` 处理 OpenCode 协议 |
| 3 | 服务端纯透传 | AI-Gateway 和 Skill Server 不解析、不转换 OpenCode 输出内容，原样中继 |
| 4 | 协议适配在客户端 | Skill 小程序负责解析 OpenCode 事件格式、渲染富文本 |
| 5 | 两个独立服务 | AI-Gateway 和 Skill Server 独立部署，通过 WebSocket 长连接传递流式数据 |
| 6 | AK/SK 认证 | PCAgent 连接 AI-Gateway 须通过聊天平台开放平台 AK/SK 认证 |
| 7 | 本次仅 OpenCode 本地 | 不考虑 Claude Code、OpenClaw、远程模式 |

### 1.3 系统边界

**构建的 4 个组件：**

| 组件 | 性质 | 运行位置 | 职责 |
|---|---|---|---|
| PC Agent 插件 | Electron Main Process 模块 | Windows PC 客户端 | 桥接本地 OpenCode ↔ AI-Gateway |
| AI-Gateway | 独立 Spring Boot 服务 | 服务端独立部署 | 连接管理、AK/SK 认证、透传中继 |
| Skill 服务端 | 独立 Spring Boot 服务 | 服务端独立部署 | 会话持久化、客户端推流、IM 消息发送 |
| Skill 客户端 | 小程序 | IM 客户端小程序容器 | "/" 触发、Mini Bar、覆盖式对话 UI |

**外部依赖（不构建）：**

- IM 聊天客户端（已有）
- IM 服务端 / 平台 API（已有）
- "/" 命令框架（已有）
- OpenCode（第三方，PC Agent 通过 SDK 连接）
- 聊天平台开放平台 AK/SK 体系（已有）

## 3. 组件设计

### 3.1 PCAgent Plugin（Electron Main Process）

**定位**：桥接本地 OpenCode 与远端 AI-Gateway，运行在 IM 客户端的 Electron Main Process 中。

**参考**：[message-bridge-opencode-plugin](https://github.com/YuanG1944/message-bridge-opencode-plugin) 架构

#### 3.1.1 模块划分

| 模块 | 文件 | 职责 |
|---|---|---|
| 插件主入口 | `PcAgentPlugin.ts` | 生命周期管理（start/stop），协调各模块 |
| AK/SK 认证 | `AkSkAuth.ts` | 用 SK 对 timestamp+nonce 签名，生成认证参数 |
| Gateway 连接 | `GatewayConnection.ts` | WebSocket 长连接管理，断线指数退避重连（1s→2s→4s，上限 30s） |
| OpenCode 桥接 | `OpenCodeBridge.ts` | `@opencode-ai/sdk` 封装：event.subscribe()、session.chat/create/list |
| 事件中继 | `EventRelay.ts` | OpenCode SSE 事件 → Gateway；Gateway 指令 → OpenCode SDK 调用 |
| 健康检测 | `HealthChecker.ts` | 定期 `api.global.health()` 检测 OpenCode 可达性 |
| 配置管理 | `config/AgentConfig.ts` | AK/SK、Gateway URL 等配置 |

#### 3.1.2 目录结构

```

├── PcAgentPlugin.ts
├── AkSkAuth.ts
├── GatewayConnection.ts
├── OpenCodeBridge.ts
├── EventRelay.ts
├── HealthChecker.ts
└── config/
    └── AgentConfig.ts
```

#### 3.1.3 核心流程伪码

```typescript
class PcAgentPlugin {
  private gateway: GatewayConnection;
  private opencode: OpenCode;

  async start(config: { ak: string; sk: string; gatewayUrl: string }) {
    // 1. AK/SK 认证 → 建立 Gateway WebSocket
    this.gateway = new GatewayConnection(config.gatewayUrl);
    await this.gateway.connect(AkSkAuth.sign(config.ak, config.sk));

    // 2. 初始化 OpenCode SDK
    this.opencode = new OpenCode({ baseURL: 'http://localhost:54321' });

    // 3. 订阅 OpenCode 事件流 → 透传给 AI-Gateway
    const stream = await this.opencode.event.subscribe();
    for await (const event of stream) {
      this.gateway.send({ type: 'tool_event', sessionId: event.sessionId, event });
    }

    // 4. 监听 AI-Gateway 指令 → 调用 OpenCode SDK
    this.gateway.onMessage(async (msg) => {
      if (msg.type === 'invoke' && msg.action === 'chat') {
        await this.opencode.session.chat(msg.payload.toolSessionId, {
          text: msg.payload.text
        });
      } else if (msg.type === 'invoke' && msg.action === 'create_session') {
        const session = await this.opencode.session.create(msg.payload);
        this.gateway.send({ type: 'session_created', session });
      }
    });
  }
}
```

#### 3.1.4 关键行为

| 行为 | 机制 |
|---|---|
| 断线重连 | 指数退避：1s → 2s → 4s → 8s → 16s → 30s（上限），重连成功后重置 |
| 事件流降级 | 主事件流 30s 无非心跳事件时切换 `api.global.event()` 备用流 |
| 健康检测 | 每 30s 调用 `api.global.health()`，不可达时上报 `agent_offline` |
| 同一工具单连接 | 同一 AK + 同一工具类型只允许一个活跃连接，新连接踢掉旧连接 |

```json

  "ws": "^8"
```

#### 3.4.3 协议适配层详解

| 文件 | 职责 | 输入 | 输出 |
|---|---|---|---|
| `OpenCodeEventParser.ts` | 识别事件类型 | 原始 OpenCode 事件 JSON | 类型化事件对象 |
| `StreamAssembler.ts` | delta 拼接 | 多个 delta 事件 | 完整消息文本 |
| `ToolUseRenderer.ts` | Tool Use 可视化 | tool.* 事件 | 可渲染的 UI 结构 |
| `types.ts` | 类型定义 | — | TypeScript 接口 |

**支持的 OpenCode 事件类型：**

- `message.updated` — 消息内容更新（含 delta）
- `session.completed` — 会话完成
- `session.created` — 会话创建
- `tool.start` — 工具调用开始
- `tool.result` — 工具调用结果
- `tool.error` — 工具调用错误

**版本兼容策略**：当 OpenCode 更新事件格式时，仅需更新 `protocol/` 模块，服务端无需改动。

## 4. 数据流

### 4.1 消息发送流（用户 → AI）

```
用户输入问题
  → Skill 小程序 POST /api/skill/sessions/{id}/messages
    → Skill Server 持久化消息
      → Skill Server 通过 WebSocket 发送到 AI-Gateway
        → AI-Gateway 通过 WebSocket 发送到 PCAgent
          → PCAgent 调用 opencode.session.chat()
            → OpenCode 处理
```

### 4.2 消息接收流（AI → 用户）

```
OpenCode SSE 事件流
  → PCAgent event.subscribe() 接收
    → PCAgent 通过 WebSocket 透传到 AI-Gateway
      → AI-Gateway 通过 WebSocket 透传到 Skill Server
        → Skill Server 持久化消息（原始事件）
          → Skill Server 通过 WebSocket 推流到 Skill 小程序
            → Skill 小程序 protocol/ 解析 → 渲染 UI
```

### 4.3 选文发送到 IM 流

```
用户选中文本 → 点击"发送到聊天"
  → Skill 小程序 POST /api/skill/sessions/{id}/send-to-im
    → Skill Server 调用 IM 平台 API
      → IM 聊天窗显示消息
```

```
Skill小程序   Skill Server   AI-Gateway   PCAgent    OpenCode
```

## 5. 通信协议定义

## 7. 认证流程

### 7.1 AK/SK 认证（PCAgent → AI-Gateway）

```
1. 管理员在聊天平台开放平台创建应用 → 获得 AK（Access Key）和 SK（Secret Key）
2. 用户在 Electron 客户端设置中配置 AK/SK
3. PCAgent 启动时：
   a. 生成 timestamp（当前时间戳）和 nonce（随机字符串）
   b. 构造签名字符串："{ak}\n{timestamp}\n{nonce}"
   c. 用 SK 作为密钥，HMAC-SHA256 计算签名
   d. 携带 ak + timestamp + nonce + signature 发起 WebSocket 握手
4. AI-Gateway 验证：
   a. 根据 AK 查询对应 SK
   b. 用同样算法计算签名，比对
   c. 检查 timestamp 防重放（±5min 窗口）
   d. 检查 nonce 防重复（Redis 存储已用 nonce，TTL 5min）
5. 验证通过 → 建立持久 WebSocket 连接
```

### 7.2 签名算法

```
signature = HMAC-SHA256(
  key = SK,
  message = "{AK}\n{timestamp}\n{nonce}"
)
```

### 7.3 内部服务认证（AI-Gateway ↔ Skill Server）

- 使用预配置的 internal token（部署时配置）
- WebSocket 握手时通过 query parameter 传递：`?token={internal_token}`

## 8. 高可用（HA）策略

### 8.1 AI-Gateway 多实例

```
              Load Balancer (L7, 支持 WS upgrade)
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
   [GW Node 1]   [GW Node 2]   [GW Node 3]
        ↕              ↕              ↕
              Redis（Agent 连接注册 + Pub/Sub 中继）
```

**Agent 连接注册（Redis）：**

```
Key:  gw:agent:conn:{agentId}
Val:  {"instanceId":"gw-node-1","userId":123,"toolType":"OPENCODE"}
TTL:  90s（心跳每 30s 刷新）
```

**跨实例中继（Redis Pub/Sub）：**

```
Channel: gw:relay:{instanceId}
```

```java
// 路由逻辑
public void routeToAgent(Long agentId, GatewayMessage msg) {
    String target = redis.get("gw:agent:conn:" + agentId);
    if (target.equals(selfInstanceId)) {
        localAgentPool.send(agentId, msg);
    } else {
        redis.publish("gw:relay:" + target, msg);
    }
}
```

### 8.2 Skill Server 多实例

Skill Server 无状态（会话数据在 MySQL，推流状态在 Redis），可直接水平扩展。

```
Key:  skill:session:subs:{sessionId}
Val:  Set<instanceId>
```

### 8.3 故障恢复

| 故障场景 | 恢复策略 |
|---|---|
| PCAgent 断连 | 指数退避重连；AI-Gateway 心跳超时推 `agent_offline` |
| AI-Gateway 单节点宕机 | LB 自动切换；PCAgent 重连到其他节点 |
| Skill Server 单节点宕机 | LB 自动切换；无状态可即时恢复 |
| AI-Gateway ↔ Skill Server 连接断开 | Skill Server 端指数退避重连；断开期间缓冲到 Redis |
| Redis 不可用 | 降级为本地路由（单实例模式） |
| MySQL 不可用 | 返回错误；消息推流不受影响（仅持久化失败） |

## 9. 协议兼容策略

### 9.1 当前实现（OpenCode 本地）

- PCAgent 通过 `@opencode-ai/sdk` 订阅 OpenCode SSE 事件流，原样透传
- AI-Gateway → Skill Server → Skill 小程序全链路透传
- Skill 小程序 `protocol/` 模块负责解析 OpenCode 事件格式

### 9.2 版本兼容

当 OpenCode 更新事件格式时：
- **仅需更新**：Skill 小程序 `protocol/` 模块
- **无需改动**：AI-Gateway、Skill Server、PCAgent

### 9.3 后续多工具扩展（不在本次范围）

新增 AI 工具时：
- Skill 小程序新增对应 `protocol/XxxParser.ts`
- PCAgent 新增对应 Bridge 模块（如 SubprocessBridge、WebSocketBridge）
- 服务端仍保持透传，无需改动
