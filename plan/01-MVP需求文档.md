# opencode-CUI — MVP 需求文档

> 版本：V2 | 日期：2026-03-02 | 关联：`00-方案设计主文档.md`

---

## 1. 需求背景

### 1.1 现状与痛点

现有聊天工具（类 DingTalk/Feishu/Slack）的 IM 消息仅支持纯文本，无法承载 AI 编码工具的富文本输出：

| 痛点 | 描述 |
|---|---|
| 无代码高亮 | AI 返回的代码块在 IM 中以纯文本显示，可读性差 |
| 无流式响应 | AI 生成内容需等待完成后一次性展示，体验延迟大 |
| 无 Markdown 渲染 | 标题、列表、表格等格式化内容无法正确展示 |
| 无 Tool Use 展示 | AI 调用工具（文件读写、终端命令等）的过程和结果无法可视化 |
| 上下文割裂 | 开发者需在 IM 和 AI 工具之间频繁切换，工作流被打断 |

### 1.2 解决方案

通过 IM 已有的 "/" 命令框架触发 **Skill 交互**——用户选择 SKILL 后，IM 聊天框上方出现微型状态条（Mini Bar），可展开为 Skill 小程序客户端，在其中与 AI 工具（OpenCode）进行多轮对话。对话结果可选中发送回 IM 聊天。

### 1.3 MVP 范围

| 维度 | MVP 范围 | 后续迭代（不在本次范围） |
|---|---|---|
| AI 工具 | **仅 OpenCode** 本地模式 | Claude Code、OpenClaw、远程模式 |
| 共享模式 | **个人私有**会话 | 团队共享 |
| 试点平台 | **Windows PC**（Electron + React + TS） | Mac、Linux、Web |
| 部署模式 | AI 工具与客户端同机（localhost） | 远程 AI 工具实例 |

## 2. 目标用户

### 2.1 用户画像

- **角色**：使用现有聊天工具（类企业 IM）的软件开发者
- **环境**：Windows PC，已安装聊天客户端（Electron），本地已安装 OpenCode
- **日常场景**：在 IM 中沟通的同时需要使用 AI 编码辅助工具
- **核心诉求**：不离开 IM 即可与 AI 编码工具交互

### 2.2 使用前提

- 用户已在本地安装并启动 OpenCode（`opencode serve`，默认 `localhost:54321`）
- 用户已获取聊天平台开放平台的 AK/SK 并在客户端中配置
- 用户所在组织已部署 AI-Gateway 和 Skill Server

## 3. 用户故事

### US-1：触发 Skill

> **作为**一名开发者，
> **我希望**在 IM 输入框输入 "/" 后直接看到 OpenCode SKILL 选项并点击触发，
> **以便**我无需离开 IM 就能启动 AI 编码对话。

**验收标准**：
- 输入 "/" 后 SKILL 列表中显示 "OpenCode" 选项（含图标和描述）
- 点击后 IM 聊天框出现一行高亮 SKILL 名称（如 "✦ OpenCode"）
- 用户可在 IM 输入框中继续输入问题

### US-2：发送问题并查看 Mini Bar

> **作为**一名开发者，
> **我希望**输入问题点击"生成"后看到处理状态，
> **以便**我知道 AI 正在处理我的请求。

**验收标准**：
- 点击"生成"后 IM 聊天区出现 1 行高的 Mini Bar
- Mini Bar 显示 "OpenCode 处理中…" 状态
- Mini Bar 右侧有"展开"按钮

### US-3：展开 Skill 小程序进行多轮对话

> **作为**一名开发者，
> **我希望**点击"展开"后能在 Skill 小程序中查看完整 AI 响应并继续对话，
> **以便**我能进行深度的多轮 AI 编码交互。

**验收标准**：
- 点击"展开"后 Skill 小程序覆盖 IM 聊天窗上方
- AI 响应以流式方式逐字显示
- 代码块有语法高亮和"复制"按钮
- 可在输入框中继续输入问题，支持多轮对话
- 支持 Shift+Enter 换行

### US-4：选文发送到 IM

> **作为**一名开发者，
> **我希望**选中 AI 回复中的任意文本并发送到 IM 聊天，
> **以便**我能将 AI 生成的代码或说明分享给同事。

**验收标准**：
- 选中 AI 回复文本时浮现"发送到聊天"按钮
- 点击后文本作为 IM 消息发送到当前聊天
- 发送成功后有提示反馈

### US-5：会话历史恢复

> **作为**一名开发者，
> **我希望**重新打开 Skill 小程序后能看到之前的对话记录，
> **以便**我能继续之前的对话上下文。

**验收标准**：
- 关闭并重新展开 Skill 小程序后，历史消息从数据库恢复
- 消息顺序正确，代码块仍有语法高亮
- 可查看和切换多个历史会话

### US-6：离线状态提示

> **作为**一名开发者，
> **我希望**在 OpenCode 未启动时能得到明确提示，
> **以便**我知道需要先启动本地 AI 工具。

**验收标准**：
- 本地 OpenCode 未启动时，Mini Bar 显示 "工具离线"
- Skill 小程序内显示离线状态指示
- OpenCode 重新启动后自动恢复连接，状态更新为在线

### US-7：无回归保证

> **作为**一名普通 IM 用户，
> **我希望**IM 的聊天功能不受 Skill 集成影响，
> **以便**我的日常沟通不被干扰。

**验收标准**：
- 不使用 Skill 时，IM 聊天功能完全正常
- Skill 小程序收起后，IM 聊天窗恢复原状
- PCAgent 插件不影响 IM 客户端启动速度和稳定性

## 5. 非功能需求

| 编号 | 维度 | 要求 |
|---|---|---|
| NFR-1 | **性能** | AI 首字节到达后 200ms 内开始渲染；消息列表支持 1000+ 消息不卡顿（虚拟滚动） |
| NFR-2 | **可用性** | AI-Gateway 和 Skill Server 支持多实例部署，单实例故障不影响整体服务 |
| NFR-3 | **延迟** | 端到端消息中继延迟（PCAgent → Skill 小程序）≤ 100ms（不含 AI 生成时间） |
| NFR-4 | **安全** | AK/SK 签名认证防重放（5min 时间窗口）；WebSocket 传输使用 WSS |
| NFR-5 | **兼容性** | 不影响现有 IM 聊天功能（零回归）；PCAgent 不影响客户端启动速度 |
| NFR-6 | **可维护性** | OpenCode 协议适配集中在 Skill 小程序 `protocol/` 模块，服务端透传不解析 |
| NFR-7 | **数据持久化** | 所有会话和消息持久化到 MySQL，支持历史查询 |
| NFR-8 | **可观测性** | 关键链路日志（连接建立/断开、消息收发、错误）；健康检查端点 |

## 7. 范围边界

### 7.1 本次范围内（In Scope）

- 4 个新建组件：PCAgent Plugin、AI-Gateway、Skill Server、Skill 小程序
- OpenCode 本地模式（localhost:54321）通过 `@opencode-ai/sdk` 集成
- AK/SK 认证机制
- "/" 命令触发 → Mini Bar → Skill 小程序 → 多轮对话 → 选文发送 IM
- 会话和消息持久化（MySQL）
- 流式渲染（Markdown + 代码高亮 + Tool Use）
- PCAgent 断线重连和健康检测
- AI-Gateway 和 Skill Server 多实例 HA 基础支持

### 7.2 本次范围外（Out of Scope）

| 排除项 | 说明 |
|---|---|
| Claude Code 集成 | 后续迭代，需 subprocess bridge |
| OpenClaw 集成 | 后续迭代，需 WebSocket RPC bridge |
| 远程模式 | AI 工具在远程机器上运行 |
| 团队共享会话 | 多人共享同一 AI 对话 |
| Mac/Linux 客户端 | 首期仅 Windows PC |
| 文件上传/下载 | 在 Skill 小程序中上传文件给 AI |
| 语音/图片输入 | 多模态输入 |
| 权限管理后台 | AK/SK 管理 UI |
| 监控告警 | Prometheus/Grafana 集成 |

### 7.3 依赖项（外部已有）

| 依赖 | 提供方 | 说明 |
|---|---|---|
| IM 聊天客户端 | 已有 | Electron 客户端，提供 "/" 框架和小程序容器 |
| IM 服务端 / 平台 API | 已有 | 提供 IM 消息发送 API |
| "/" 命令框架 | 已有 | 提供 SKILL 注册和回调机制 |
| 聊天平台开放平台 AK/SK | 已有 | 提供应用认证体系 |
| OpenCode | 第三方 | 本地 AI 编码工具，提供 REST + SSE API |
| `@opencode-ai/sdk` | 第三方 | TypeScript SDK，v1.2+ |
