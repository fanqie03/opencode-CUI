# OpenCode-CUI 架构调整说明

**版本**: v1.0
**日期**: 2026-03-04
**状态**: 已确认
**关联文档**: `需求澄清文档.md`, `plan/00-方案设计主文档.md`, `plan/02-架构设计文档.md`

---

## 一、调整概述

基于 2026-03-04 的需求澄清结果，项目架构从"IM 平台集成方案"调整为"本地联调 Demo 优先"策略。

### 核心变化

| 维度 | 原方案 | 调整后方案 |
|------|---------|------------|
| **部署目标** | IM 平台集成（PCAgent 插件 + Skill 小程序） | 本地联调 Demo（独立 PCAgent + Web UI） |
| **客户端形态** | IM 客户端插件 + IM 小程序 | 独立进程 + React Web 应用 |
| **开发优先级** | 完整功能 → IM 集成 | 核心链路验证 → 功能完善 → IM 集成 |
| **验证目标** | 端到端 IM 集成 | 技术链路可行性 + 架构扩展性 |

## 三、组件调整详情

### 3.1 PCAgent

#### 原方案
- **运行方式**: IM 客户端 Electron 插件
- **生命周期**: 随 IM 客户端启动/关闭
- **配置方式**: IM 客户端设置界面
- **依赖**: IM 客户端进程

#### 调整后方案
- **运行方式**: 独立 Node.js 进程
- **生命周期**: 独立启动/关闭，不依赖 IM 客户端
- **配置方式**: 命令行参数或配置文件
- **依赖**: 仅依赖 OpenCode 和 AI-Gateway

#### 实施建议
1. 将 PCAgent 改造为独立的 Node.js 应用
2. 提供命令行启动脚本：`npm start -- --ak=xxx --sk=xxx --gateway=ws://localhost:8080`
3. 支持配置文件：`config.json` 或环境变量
4. 实现健康检查和自动重启机制
5. 提供日志输出和错误诊断

---

### 3.2 Skill 客户端

#### 原方案
- **运行环境**: IM 客户端小程序容器
- **触发方式**: "/" 命令框架
- **UI 形态**: Mini Bar + 覆盖式对话窗口
- **集成方式**: 注册到 IM 平台的 SKILL 列表

#### 调整后方案
- **运行环境**: 独立 Web 应用（浏览器）
- **触发方式**: 直接访问 URL（如 `http://localhost:3000`）
- **UI 形态**: 完整的 Web 页面
- **集成方式**: 无需 IM 平台集成

#### 实施建议
1. 使用 React + TypeScript 开发
2. 使用 `create-react-app` 或 `vite` 快速搭建
3. 复用 `skill-miniapp` 目录中的协议适配层代码
4. 实现以下核心功能：
   - WebSocket 连接管理
   - 消息列表渲染（支持流式更新）
   - Markdown 渲染（使用 `react-markdown`）
   - 代码高亮（使用 `shiki` 或 `prism`）
   - Tool Use 可视化
   - 会话历史加载

---

### 3.3 AI-Gateway

#### 变化
- **无变化**：AI-Gateway 的职责和实现保持不变
- **配置调整**：支持本地部署和服务端部署的配置切换

#### 实施建议
1. 验证现有代码的完整性
2. 添加配置文件支持（`application-local.yml` 和 `application-server.yml`）
3. 实现健康检查接口（`/actuator/health`）
4. 添加详细的日志输出

#### 变化
- **无变化**：Skill Server 的职责和实现保持不变
- **IM 消息发送**: Demo 阶段暂不实现，保留接口

### 3.5 新增：AK/SK 管理后台

#### 功能范围
- 用户注册/登录（简单的用户名/密码）
- AK/SK 的 CRUD 操作
- 查看 AK/SK 的使用统计（可选）
- 权限验证

#### 技术栈
- **后端**: Spring Boot + Spring Security
- **前端**: React + Ant Design
- **数据库**: MySQL 5.7

#### 数据库表设计
```sql
-- 用户表
CREATE TABLE user (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    password VARCHAR(255) NOT NULL,  -- BCrypt 加密
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- AK/SK 表
CREATE TABLE access_key (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    access_key VARCHAR(32) UNIQUE NOT NULL,  -- 随机生成
    secret_key VARCHAR(64) NOT NULL,         -- 随机生成
    description VARCHAR(255),
    status ENUM('ACTIVE', 'DISABLED') DEFAULT 'ACTIVE',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES user(id) ON DELETE CASCADE
);

-- 使用记录表（可选）
CREATE TABLE access_key_usage (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    access_key_id BIGINT NOT NULL,
    event_type VARCHAR(50),  -- 'LOGIN', 'MESSAGE_SENT', etc.
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (access_key_id) REFERENCES access_key(id) ON DELETE CASCADE
);
```

#### 实施建议
1. 创建独立的 Spring Boot 模块：`aksk-admin`
2. 实现 RESTful API：
   - `POST /api/auth/register` - 用户注册
   - `POST /api/auth/login` - 用户登录
   - `GET /api/keys` - 查看 AK/SK 列表
   - `POST /api/keys` - 创建 AK/SK
   - `DELETE /api/keys/{id}` - 删除 AK/SK
3. 实现前端页面：
   - 登录页
   - AK/SK 管理页（列表 + 创建 + 删除）
4. 集成到 AI-Gateway 的 AK/SK 验证逻辑

```
│  └─────────────┘  └─────────────┘  └─────────┘      │

└─────────────────┘
```

**启动顺序**:
1. MySQL + Redis
2. AI-Gateway + Skill Server
3. AK/SK Admin（创建测试用户和 AK/SK）
4. PCAgent（使用创建的 AK/SK）
5. Web UI

**一键启动脚本**:
```bash
#!/bin/bash
# start-demo.sh

# 1. 启动基础服务
docker-compose up -d mysql redis

# 2. 启动后端服务
cd ai-gateway && mvn spring-boot:run -Dspring-boot.run.profiles=local &
cd skill-server && mvn spring-boot:run -Dspring-boot.run.profiles=local &
cd aksk-admin && mvn spring-boot:run -Dspring-boot.run.profiles=local &

# 3. 等待后端服务启动
sleep 10

# 4. 启动 PCAgent
cd pc-agent && npm start -- --config=config.local.json &

# 5. 启动前端
cd web-ui && npm start &
cd aksk-admin-ui && npm start &

echo "Demo started! Visit http://localhost:3000"
```

**Nginx 配置示例**:
```nginx
upstream ai_gateway {
    server localhost:8080;
}

upstream skill_server {
    server localhost:8081;
}

upstream aksk_admin {
    server localhost:8082;
}

server {
    listen 80;
    server_name test-server.com;

    # Web UI 静态文件
    location / {
        root /var/www/web-ui;
        try_files $uri /index.html;
    }

    # AI-Gateway WebSocket
    location /gateway {
        proxy_pass http://ai_gateway;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
     proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }

    # Skill Server WebSocket
    location /skill {
        proxy_pass http://skill_server;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }

    # AK/SK Admin API
    location /admin {
        proxy_pass http://aksk_admin;
        proxy_set_header Host $host;
    }
}
```

## 七、总结

本次架构调整的核心思想是**降低外部依赖，先验证技术可行性**。通过开发本地联调 Demo，我们可以：

1. **快速验证核心技术链路**: OpenCode → PCAgent → AI-Gateway → Skill Server → Web UI
2. **验证架构的扩展性**: 多实例协调、透明中继、协议适配层
3. **降低 IM 集成的风险**: 先验证技术可行性，再与 IM 平台团队对接
4. **提供可演示的原型**: 便于向利益相关方展示项目进展

调整后的架构保留了原方案的核心设计原则（透明中继、双服务分离、多实例协调），只是将客户端形态从"IM 插件 + 小程序"调整为"独立进程 + Web 应用"。这种调整是**可逆的**，后续可以平滑地迁移到 IM 平台集成方案。
