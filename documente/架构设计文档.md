## 一、架构概览

### 1.1 系统定位

OpenCode-CUI 是一个**分布式事件驱动架构**系统，采用**透明中继模式**将本地 AI 编码工具（OpenCode）的富文本输出（代码块、Markdown、流式响应、Tool Use 等）通过 WebSocket 长连接和 Redis Pub/Sub 传递到用户界面。

项目分两阶段实施：
- **Demo 阶段**：独立 Web UI + 独立 PCAgent 进程，验证核心技术链路
- **最终实现**：IM 平台集成（PCAgent 插件 + Skill 小程序），支持多工具、团队共享

### 1.2 设计原则

1. **透明中继**：AI-Gateway 和 Skill Server 不解析 OpenCode 事件内容，原样透传
2. **协议适配在客户端**：Skill 客户端（Web UI / 小程序）负责解析 OpenCode 事件格式、渲染富文本
3. **双服务分离**：AI-Gateway（连接管理 + 认证）与 Skill Server（业务逻辑 + 持久化）独立部署
4. **多实例协调**：通过 Redis Pub/Sub 实现无状态水平扩展
5. **AK/SK 认证**：PCAgent 连接 AI-Gateway 须通过 HMAC-SHA256 签名认证
6. **单设备登录**：同一 AK + 同一工具类型仅允许一个活跃连接，新连接踢掉旧连接
7. **离线容忍**：OpenCode 离线时消息排队，上线后自动发送

### 1.3 技术选型总览

| 层级 | 组件 | 选型 | 版本 | 理由 |
|------|------|------|------|------|
| 后端框架 | Spring Boot | 3.4.6 | Java 21 | 虚拟线程、生态完善、长期支持 |
| 数据库 | MySQL | 5.7 | — | 团队熟悉、稳定可靠 |
| ORM | MyBatis | 3.0.4 | — | 灵活 SQL 映射、复杂查询友好 |

## 四、组件详解

### 4.1 PCAgent

#### Demo 阶段：独立 Node.js 进程

**启动方式**：
```bash
npm start -- --ak=your_ak --sk=your_sk --gateway=ws://localhost:8081/ws/agent
# 或使用配置文件
npm start -- --config=config.local.json
```

**配置文件** (`config.local.json`)：
```json
{
  "ak": "your_access_key",
  "sk": "your_secret_key",
  "gatewayUrl": "ws://localhost:8081/ws/agent",
  "opencodeUrl": "http://localhost:54321",
  "heartbeatInterval": 30000,
  "reconnectMaxRetries": 3
}
```

#### 最终阶段：Electron 插件

- 随 IM 客户端启动，在独立子进程中运行
- 配置通过 IM 客户端设置界面
- 崩溃自动重启，不影响 IM 主进程

#### 目录结构

```
src/main/pc-agent/
├── PcAgentPlugin.ts          插件主入口（生命周期管理）
├── AkSkAuth.ts               AK/SK 签名生成
├── GatewayConnection.ts      AI-Gateway WebSocket 连接管理
├── OpenCodeBridge.ts         @opencode-ai/sdk 桥接
├── EventRelay.ts             事件双向中继
├── HealthChecker.ts          OpenCode 健康检测
└── config/
    └── AgentConfig.ts        配置管理
```

#### 核心伪码

```typescript
class PcAgentPlugin {
  private gateway: GatewayConnection;
  private opencode: OpenCode;

  async start(config: AgentConfig) {
    // 1. AK/SK 认证 → 建立 Gateway WebSocket
    this.gateway = new GatewayConnection(config.gatewayUrl);
    await this.gateway.connect(AkSkAuth.sign(config.ak, config.sk));

    // 2. 初始化 OpenCode SDK
    this.opencode = new OpenCode({ baseURL: config.opencodeUrl });

    // 3. 订阅 OpenCode 事件流 → 透传给 AI-Gateway
    const stream = await this.opencode.event.subscribe();
    for await (const event of stream) {
      this.gateway.send({
        type: 'tool_event',
        sessionId: event.sessionId,
        event
      });
    }

    // 4. 监听 AI-Gateway 指令 → 调用 OpenCode SDK
    this.gateway.onMessage(async (msg) => {
      if (msg.type === 'invoke' && msg.action === 'chat') {
        await this.opencode.session.chat(msg.payload.toolSessionId, {
          text: msg.payload.text
        });
      }
    });
  }
}
```

### 4.2 AI-Gateway (端口 8081)

**职责**：连接管理、AK/SK 认证、事件透传、多实例协调

#### 关键类

| 类 | 职责 |
|----|------|
| `AgentWebSocketHandler` | PCAgent WebSocket 连接生命周期管理 |
| `AkSkAuthService` | HMAC-SHA256 签名验证、时间戳窗口、Nonce 防重放 |
| `AgentRegistryService` | Agent 注册、心跳、上下线状态管理 |
| `EventRelayService` | PCAgent ↔ Skill Server 双向消息中继 |
| `SkillServerWSClient` | → Skill Server WebSocket 客户端 |
| `RedisMessageBroker` | Redis Pub/Sub 跨实例路由 |

#### WebSocket 端点

```
ws://{host}:8081/ws/agent?ak={ak}&ts={timestamp}&nonce={nonce}&sign={signature}
```

#### REST API

```
GET  /api/gateway/agents              查询在线 PCAgent 列表
GET  /api/gateway/agents/{id}/status  查询 Agent 状态
POST /api/gateway/agents/{id}/invoke  下发指令给 PCAgent (备用通道)
GET  /actuator/health                 健康检查
```

#### 数据库表


```
- 定时任务每 10 分钟: ACTIVE → IDLE (30 分钟未活动)
- 用户手动关闭: → CLOSED
- 并发限制: 单用户最多 10 个 ACTIVE 会话
```

### 5.3 Redis Pub/Sub 通道设计

| 通道 | 发布者 | 订阅者 | 用途 |
|------|--------|--------|------|
| `agent:{agentId}` | Skill Server | Gateway | 跨实例路由 invoke 命令 |
| `session:{sessionId}` | Gateway | Skill Server | 跨实例路由会话消息 |
| `gw:relay:{instanceId}` | Skill Server | 特定 Gateway 实例 | 定向路由 |

**Agent 连接注册**：
```
Key:  gw:agent:conn:{agentId}
Val:  {"instanceId":"gw-node-1","userId":123,"toolType":"OPENCODE"}
TTL:  90s (心跳每 30s 刷新)
```

### 5.4 序列号与乱序检测

- 每个会话独立计数，从 1 开始
- 小间隙 (1-5)：记录警告，继续处理
- 中间隙 (6-20)：请求恢复
- 大间隙 (>20)：触发重连

### 5.5 REST API 响应规范

```json
// 成功
{"code": 0, "message": "success", "data": {...}}

// 错误
{"code": 401, "message": "AK/SK 认证失败", "data": null}
```

**错误码**：0 成功 | 400 参数错误 | 401 认证失败 | 403 权限不足 | 404 资源不存在 | 429 请求过频 | 500 服务器错误

## 六、数据流分析

### 6.1 上行流：用户消息 → OpenCode

```
1. 用户在 Web UI / Skill 小程序输入消息
2. POST /api/skill/sessions/{id}/messages
3. Skill Server 持久化消息到 MySQL (role=USER)
4. GatewayRelayService 发送 invoke 到 AI-Gateway
   → Redis publish agent:{agentId} (跨实例)
   → 或 WebSocket 直发 (同实例)
5. AI-Gateway EventRelayService 接收
6. 通过 WebSocket 发送到 PCAgent
7. PCAgent 调用 opencode.session.chat()
8. OpenCode 处理请求
```

### 6.2 下行流：OpenCode 响应 → 用户

```
1. OpenCode 生成 SSE 事件 (content_block_delta, tool_use 等)
2. PCAgent event.subscribe() 接收
3. WebSocket 发送 tool_event 到 AI-Gateway
4. AI-Gateway EventRelayService 透传到 Skill Server
   → Redis publish session:{sessionId} (跨实例)
5. Skill Server GatewayRelayService 接收
6. SkillMessageService 持久化 (role=ASSISTANT)
7. SkillStreamHandler 推流到 Web UI / Skill 小程序
8. 客户端 protocol/ 模块解析事件
9. UI 渲染 (Markdown + 代码高亮 + Tool Use)
```

### 6.3 时序图

```

  │               │              │           │           │
  │─ POST msg ───►│              │           │           │
  │               │─ save msg ───│           │           │
  │               │─ WS invoke ─►│           │           │
  │               │              │─ WS send ►│           │

  │               │              │◄── WS ────│           │
```

## 七、核心设计模式

### 7.1 透明中继模式

Gateway 不解析或修改消息内容，只负责转发和路由：

```java
// EventRelayService.relayToSkillServer()
GatewayMessage forwarded = message.withAgentId(agentId);
skillServerRelay.sendToSkillServer(forwarded);
// 消息内容完全不变，只添加 agentId 元数据
```

**优势**：协议独立（支持多 IDE 工具）、版本兼容（新事件类型自动支持）、低耦合。

### 7.2 AK/SK 认证与防重放

```
1. 客户端: signature = HMAC-SHA256(SK, "{AK}\n{timestamp}\n{nonce}")
2. 携带 ak, ts, nonce, sign 发起 WebSocket 握手
3. Gateway 验证:
   a. timestamp 在 ±5 分钟内
   b. nonce 未使用过 (Redis SET NX, TTL 5 分钟)
   c. 签名正确 (常数时间比较, 防时序攻击)
```

## 十、故障恢复

| 故障场景 | 症状 | 恢复策略 |
|----------|------|----------|
| PCAgent 断连 | 用户无法发送消息 | 指数退避重连 (1s→2s→4s), 最多 3 次; 超时推 agent_offline |
| Gateway 单节点宕机 | 部分用户连接断开 | LB 自动切换; PCAgent 重连其他节点 |
| Skill Server 宕机 | 推流中断 | LB 切换; 无状态可即时恢复 |
| Gateway↔SS 断开 | 消息无法中继 | Skill Server 指数退避重连; 缓冲到 Redis |
| Redis 不可用 | 多实例协调失败 | 降级为本地路由 (单实例模式) |
| MySQL 不可用 | 持久化失败 | 返回错误; 推流不受影响 |
| OpenCode 离线 | AI 无法响应 | 显示离线状态; 消息排队等待上线 |

## 十一、技术决策分析

### 11.1 为什么选择透传而非服务端解析

| 方案 | 优点 | 缺点 |
|------|------|------|
| **透传** (选定) | 协议独立、版本兼容、低耦合 | 无法在服务端统一处理 |
| 解析 | 统一处理、易于监控 | 协议绑定、版本维护复杂 |

### 11.2 为什么需要两个独立服务

| 方案 | 优点 | 缺点 |
|------|------|------|
| **双服务** (选定) | 职责清晰、独立扩展 | 部署复杂、通信开销 |
| 单服务 | 简单、部署少 | 职责混乱、扩展困难 |

Gateway 处理 I/O 密集（连接管理），Skill Server 处理计算密集（持久化、推流），分离后可独立扩展。

### 11.3 为什么 Demo 先用 Web UI

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Web UI 优先** (选定) | 无外部依赖、快速验证 | 需要后续改造 |
| 直接 IM 集成 | 一步到位 | 依赖 IM 团队、风险高 |

### 11.4 为什么使用 Redis Pub/Sub

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Redis Pub/Sub** (选定) | 简单、低延迟 | 消息不持久化 |
| RabbitMQ | 持久化、可靠 | 复杂、开销大 |
| Kafka | 高吞吐、持久化 | 复杂、延迟高 |

消息不需持久化（有 WebSocket 直连和数据库备份），低延迟是核心需求。
