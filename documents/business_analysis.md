# OpenCode-CUI 业务规则

---

## 一、系统定位

- OpenCode-CUI 对接多种交互场景（IM 消息、会议、云文档等），不负责客户端
- `skill-miniapp` 是 IM 平台中嵌入的技能小程序，用于联调测试
- **技能与数字分身的关系**：技能（skill）是数字分身（Agent）的能力
- **群聊两种模式并存**：同一个群可以同时使用「技能小程序」（场景一）和「数字分身进群」（场景二）两种交互方式
- **系统范围**：skill-server 和 ai-gateway 是我们的核心组件，IM 平台（客户端+服务端）和 skill-miniapp 前端属于外部系统

### 关键前提（场景二与单聊适用）

> 以下前提适用于场景二（数字分身进群）和单聊场景，场景一（技能小程序）不受影响。

1. **@感知由 IM 负责**：群聊中 @数字分身时，skill-server 不直接感知。IM 服务端识别到 @事件后，调用 skill-server 的 REST 接口发送消息。
2. **IM 不调会话创建接口**：IM 为每个单聊/群聊定义一个唯一标识字符串（`chatId`），通过发送消息接口传过来。skill-server 不对外暴露会话创建 API。
3. **发消息接口自动建会话**：如果 `chatId` 对应的 WelinkSession 不存在，skill-server 的发送消息接口需要**自动创建** WelinkSession 并绑定 ToolSession，然后再处理消息。
4. **IM 不感知 AK**：IM 服务端只知道机器人的 `welinkId`（IM 平台标识），不知道 AK。skill-server 需要调用第三方接口将 `welinkId` 解析为对应的 `ak`。
5. **群内成员管理由 IM 负责**：哪些数字分身在哪些群里，由 IM 平台管理，不在 skill-server 范围内。skill-server 只需校验 welinkId 能否找到对应的 ak、以及 chatId 是否有对应会话。
6. **会话标识三元组**：会话通过 `chat_domain + chat_id + ak` 三元组唯一标识。`chat_domain` 表示来源场景（`miniapp`/`im`/`meeting`/`doc`），支持未来扩展到会议、云文档等场景而无需变更 DB Schema。

## 二、群聊场景

群聊支持两种交互模式，可在同一个群中共存：

| 维度                 | 场景一：技能小程序     | 场景二：数字分身进群 |
| -------------------- | ---------------------- | -------------------- |
| 数字分身是否为群成员 | ❌ 不在群成员列表       | ✅ 出现在群成员列表中 |
| 触发方式             | 输入 `/` 唤起技能弹窗  | `@数字分身名`        |
| 对话空间             | 小程序内私密对话       | 群聊中公开对话       |
| 回复可见性           | 仅用户可见，选择性发送 | 所有群成员可见       |

---

### 场景一：技能小程序模式（数字分身不进群）

#### 场景一规则

**规则 G1：单 WS 连接**
一个客户端与 skill-server 最多一条 WS 连接，所有群/单聊事件走同一通道，通过 `sessionId` 区分。

**规则 G2：多助手多技能**
群里的技能来自多个助手，不同助手的技能可能重叠。一个技能对应一个 AK（一对一）。

**规则 G3：群+Agent 会话唯一**
同一用户在同一个群里，同一个 Agent（AK）最多只有一个活跃的 WelinkSession。WelinkSession 绑定一个 ToolSession，后续可支持更换绑定。

**规则 G4：聊天记录按群+Agent 隔离**
不同群隔离，同一群的不同 Agent 隔离。隔离维度：`(imGroupId, ak)`。

**规则 G5：会话持久化与历史恢复**
小程序收起后会话仍然存在。再次打开时继续上次会话，展示历史聊天记录。同一用户在同一群聊中再次 `/` 选择相同技能，进入之前的对话历史，不新建。

**规则 G6：技能列表来源**
技能列表由第三方服务上报，携带 AK 和技能名称。群里能用哪些技能没有限制，用户把 AK 配置到了哪些 Agent，就展示那些 Agent 的技能。

**规则 G7：AK 自动解析**
当技能没有 AK 时，客户端不传 AK，skill-server 需要：
1. 调用第三方 `POST /appstore/wecodeapi/open/user/identities` 获取用户名下所有 AK
   - 入参：`{ "userId": "d00123456" }`（小写字母+8位数字）
   - 出参：`{ "code": "200", "data": ["ak1", "ak2"] }`
2. 查询 gateway 当前用户在线的 Agent 对应的 AK
3. 两个列表取交集，选择一个在线的 AK
4. 调用第三方 `POST /appstore/wecodeapi/open/user/ak/check` 校验 AK 与用户关系
   - 入参：`{ "userId": "d00123456", "ak": "xxx" }`
   - 出参：`{ "code": "200", "data": { "checkResult": true } }`
5. 校验通过后绑定到 session

**规则 G8：发送到群聊的格式**
当前群聊只支持纯文本。AI 返回的代码块、Markdown 格式需转换和清理，保证可读性。

**规则 G9：统一会话模型**
群聊和单聊共用一套会话模型，通过 `chat_domain + chat_type + chat_id` 三元组标识会话来源。`chat_domain` 区分场景（miniapp/im/meeting/doc），`chat_type` 区分聊天类型（group/direct）。

#### 场景一用户旅程

| 步骤  | 用户行为 / 系统响应                                                                     |  关联规则  |
| :---: | --------------------------------------------------------------------------------------- | :--------: |
|   1   | 用户在 IM 群聊输入 `/`                                                                  |            |
|   2   | IM 弹出技能选择弹窗，展示当前用户可用的技能列表                                         |   G2, G6   |
|   3   | 选择技能，输入框出现胶囊，输入问题，回车                                                |            |
|   4   | IM 打开小程序 WebView（skill-miniapp），通过唯一 WS 通道通信                            |     G1     |
|   5   | 小程序请求创建/复用会话：有 AK 时查询已有会话，复用或新建；无 AK 时先走 AK 自动解析流程 | G3, G5, G7 |
|   6   | AI 处理请求，流式结果通过 WS 推送到小程序，聊天记录按群+Agent 隔离存储                  |   G1, G4   |
|   7   | 用户在小程序内与 AI 私密对话，可持续多轮；收起小程序后会话仍保持，再次打开恢复历史      |     G5     |
|   8   | 用户选中一条 AI 回复，点击「发送到群聊」                                                |            |
|   9   | AI 回复经过格式清理（Markdown 转纯文本），发送到 IM 群聊                                |     G8     |
|  10   | 群聊中展示带「来自 XX助手 XX技能」标记的消息                                            |     G9     |

### 场景二：数字分身进群模式

#### 场景二规则

**规则 GA1：添加入口** `[IM 侧]`
群聊标题栏右侧有「邀请成员」（+）按钮，点击弹出「邀请成员」弹窗。弹窗 tab 栏包含「助手」选项卡，列出当前用户名下**已连接**且**未在本群中**的数字分身。

**规则 GA2：进群流程** `[IM 侧]`
用户在「助手」tab 中勾选数字分身，点击「确定」后：
- 数字分身加入群聊，出现在群成员列表的「助手」分区中（带「助手」标签和独立头像）
- 群聊中插入系统消息：「XX 已加入群聊」
- 一个群可以添加**多个**数字分身

**规则 GA3：@触发对话**
用户在群聊输入框中 `@数字分身名` 并发送消息，IM 服务端感知 @事件后调用 skill-server REST 接口。数字分身回复时需 `@发送者`（@mention 能力由 IM 提供，skill-server 只需在调用 IM 发消息接口时将发送者信息填入对应字段），回复消息在群聊中公开展示。

**规则 GA4：群聊公开回复**
数字分身的回复直接显示在群聊消息流中，所有群成员可见。回复消息带有蓝色渐变气泡和「助手」标签，以区分普通用户消息。

**规则 GA5：会话独立**
数字分身进群模式的会话与场景一（技能小程序模式）的会话相互独立，各自维护独立的上下文。

**规则 GA5a：上下文超限自动重建**
当对话的上下文长度超过限制时，需要自动新建 ToolSession 并重新绑定到 WelinkSession，以保证后续对话正常进行。

**规则 GA6：成员列表展示** `[IM 侧]`
进群的数字分身在群成员列表中独立展示，位于「助手」分区（与普通成员分区隔开），显示头像、名称和「助手」标签。

**规则 GA7：通信架构**
场景二使用 IM 客户端和 IM 服务端进行消息收发，skill-server 通过 REST 接口与 IM 服务端交互：
- **接收消息**：IM 服务端感知到 @数字分身事件后，调用 skill-server 的 REST 接口发送消息，传入 `chatId`（群聊标识）+ `welinkId`（机器人 IM 标识）+ 消息内容 + 群聊历史（可选）+ 发送者信息
- **welinkId → ak 解析**：IM 不感知 AK，skill-server 需调用第三方接口将 `welinkId` 解析为对应的 `ak`（与场景一 G7 类似的第三方接口调用）
- **自动会话管理**：skill-server 用 `chat_domain('im') + chatId + ak` 三元组查找已有 session，找不到则**自动创建** WelinkSession（关联 welinkId）并绑定 ToolSession
- **发送回复**：Agent 输出完成后，skill-server 通过 IM 服务端的**单聊接口**将结果发送到群聊
- **长连接能力**：skill-server 集群与 IM 服务端集群之间需具备长连接通信能力，长连接采用 **Token 认证**（非 Cookie 认证）

**规则 GA8：群聊上下文注入**
IM 在 @数字分身时，会通过接口参数将**之前的群聊对话历史**一并传递。skill-server 需要：
- 支持**可配置的提示词模板**，将群聊历史会话和当前 @对话组合成一个完整的 Prompt 发送给 Agent
- **群聊不做消息持久化**：聊天记录由 IM 平台管理，skill-server 不存储群聊场景的消息

**规则 GA9：图片消息支持**
群聊场景需要支持图片消息的处理：
- **接收图片**：用户在群聊中 @数字分身时可以附带图片，skill-server 需能够接收并传递给 Agent
- **返回图片**：Agent 的回复中可以包含图片，skill-server 需能够将图片通过 IM 接口发送到群聊

**规则 GA10：IM 消息格式约束**
IM 仅支持**纯文本**和**图片**两种消息格式。Agent 返回的其他格式（Markdown、Question 卡片、Tool 调用、Permission 请求等）需要在 skill-server 侧进行转换或过滤，确保发送到 IM 的消息符合格式要求。

#### 场景二用户旅程

| 步骤  | 用户行为 / 系统响应                                                                                                   |         关联规则         |
| :---: | --------------------------------------------------------------------------------------------------------------------- | :----------------------: |
|   1   | 用户在群聊标题栏点击「邀请成员」（+）按钮                                                                             |       GA1 `[IM侧]`       |
|   2   | 弹出「邀请成员」弹窗，切换到「助手」tab，展示数字分身列表                                                             |       GA1 `[IM侧]`       |
|   3   | 勾选一个或多个数字分身，点击「确定」                                                                                  |       GA2 `[IM侧]`       |
|   4   | 群聊中出现系统消息「XX 已加入群聊」，成员列表「助手」分区出现该分身                                                   |    GA2, GA6 `[IM侧]`     |
|   5   | 用户在输入框输入 `@数字分身名` 加消息内容（可附带图片），发送                                                         |         GA3, GA9         |
|   6   | IM 服务端感知 @事件，调用 skill-server REST 接口（传入 chatId + welinkId + 消息 + 群聊历史）                          |         GA7, GA8         |
|   7   | skill-server 将 welinkId 解析为 ak，用 chatId + ak 查找/自动创建会话，通过 prompt 模板注入上下文后发送给 Agent        |         GA7, GA8         |
|   8   | Agent 回复经 skill-server 转换为纯文本/图片格式，通过 IM 单聊接口发到群聊，消息带蓝色气泡和「助手」标签，并 `@发送者` | GA3, GA4, GA7, GA9, GA10 |
|   9   | 群聊不做消息持久化，聊天记录由 IM 平台管理                                                                            |           GA8            |
|  10   | 群内所有成员均可看到数字分身的回复                                                                                    |           GA4            |
|  11   | 此对话与场景一的技能小程序会话互相独立                                                                                |           GA5            |

## 三、单聊场景

### 交互模型

单聊是直接与**数字分身**对话，不经过技能弹窗或小程序：

| 群聊场景一（技能小程序） | 群聊场景二（分身进群） | 单聊             |
| ------------------------ | ---------------------- | ---------------- |
| `/` 唤起技能弹窗         | `@数字分身名`          | 直接输入对话     |
| 小程序内私密对话         | 群聊中公开对话         | 1:1 私密对话     |
| 多个技能可选             | 多个分身可添加         | 固定一个数字分身 |
| 选择性发送到群聊         | 所有人可见             | 仅双方可见       |

### 单聊规则

**规则 D1：数字分身创建入口** `[IM 侧]`
IM 顶部 "+" 按钮的下拉菜单中，包含"创建数字分身"选项（与创建群聊、加入会议等并列）。点击后弹出创建弹窗。

**规则 D2：创建弹窗** `[IM 侧]`
弹窗内用户选择头像（预设多种风格）并输入分身名称，确认后系统在后台注册 Agent 并初始化 AK/SK。

**规则 D3：引导卡片** `[IM 侧]`
创建完成后，IM 左侧会话栏出现新的数字分身条目（显示"等待接入 AK/SK..."）。进入聊天界面后，展示引导卡片，包含 AK/SK 接入的指导步骤（获取凭证 → 绑定平台 → 开始对话），引导用户将 AK/SK 绑定到 OpenCode。

**规则 D4：连接状态展示** `[IM 侧]`
AK/SK 绑定完成后，聊天框标题栏的数字分身名称旁边显示"已连接"胶囊（绿色圆点+文字），实时指示 Agent 在线状态。侧边栏的数字分身头像右下角也有绿色在线指示器。未绑定时显示"待配置"/未连接时显示"未连接"。

**规则 D5：通信架构**
单聊使用 IM 客户端和 IM 服务端进行消息收发，skill-server 通过 REST 接口与 IM 服务端交互（与群聊场景二 GA7 一致）：
- **接收消息**：IM 服务端调用 skill-server 的 REST 接口发送消息，传入 `chatId`（单聊标识）+ `welinkId`（机器人 IM 标识）+ 消息内容 + 发送者信息
- **welinkId → ak 解析**：IM 不感知 AK，skill-server 需调用第三方接口将 `welinkId` 解析为对应的 `ak`
- **自动会话管理**：skill-server 用 `chat_domain('im') + chatId + ak` 三元组查找已有 session，找不到则**自动创建** WelinkSession（关联 welinkId）并绑定 ToolSession（单聊无群聊历史上下文注入）
- **发送回复**：Agent 输出完成后，skill-server 通过 IM 服务端的**单聊接口**将结果发送给用户
- **长连接能力**：skill-server 集群与 IM 服务端集群之间需具备长连接通信能力，长连接采用 **Token 认证**（非 Cookie 认证）

**规则 D6：创建新会话**
单聊输入框上方有一个"创建新会话"功能胶囊按钮。点击后：
- 原有聊天记录**保留**在界面上，不清空
- 聊天记录框中出现一条居中的系统气泡，提示"已创建新会话"
- 后台新建一个 ToolSession 并绑定到当前 WelinkSession（替换旧的 ToolSession）
- 后续对话走新的 ToolSession 上下文

**规则 D7：会话唯一性与 ToolSession 重建**
一个数字分身有且仅有一个 WelinkSession。该 WelinkSession 仅支持绑定一个 ToolSession。ToolSession 可通过以下方式重建：
- **手动重建**：用户点击「创建新会话」（规则 D6）
- **自动重建**：当对话上下文长度超过限制时，自动新建 ToolSession 并重新绑定，保证后续对话正常进行

**规则 D8：聊天记录跨场景隔离**
同一个数字分身，在每个群聊之间和单聊的聊天记录都互相隔离。即：群A的记录、群B的记录、单聊的记录，三者互不可见。

**规则 D9：会话持久化与历史恢复**
单聊关闭后，再次打开时展示上次的聊天记录，沿用已有会话，不新建。

**规则 D10：IM 消息格式约束与图片支持**
IM 仅支持**纯文本**和**图片**两种消息格式（与群聊场景二 GA10 一致）。Agent 返回的其他格式（Markdown、Question 卡片、Tool 调用、Permission 请求等）需要在 skill-server 侧进行转换或过滤，确保发送到 IM 的消息符合格式要求。

**规则 D11：未读消息与会话列表** `[IM 侧]`
单聊的未读消息管理（红色气泡角标、最近消息预览、标记已读/未读）和数字分身连接状态展示由 IM 平台负责，不在 skill-server 范围内。

**规则 D12：数字分身详情面板** `[IM 侧]`
标题栏最右边三个点（⋯）按钮，点击后右侧滑出详情面板，内容包括：
- 助手头像和连接状态
- 助手名称（可编辑，**修改后同步后台**）
- 描述
- API 凭证（AK/SK）：默认脱敏显示，支持显隐切换和复制
- 创建日期

**规则 D13：统一会话模型**
单聊会话使用同一套会话模型（规则 G9），通过 `chat_domain + chat_type + chat_id` 三元组标识，未来可扩展到会议、云文档等场景。
