# opencode-CUI — Skill 对话集成方案（V2）

## Context

现有聊天工具（类 DingTalk/Feishu/Slack）的 IM 消息仅支持纯文本，无法承载 AI 编码工具的富文本输出（代码块、Markdown、流式响应等）。

**V2 方案**：不新建独立 AI 对话窗口，改为通过 IM 已有的 "/" 命令框架触发 **Skill 交互**——用户选择 SKILL 后，IM 聊天框上方出现微型状态条，可展开为 Skill 小程序客户端，在其中与 AI 工具进行多轮对话。对话结果可选中发送回 IM 聊天。

客户端集成 **PCAgent 插件**（参考 [message-bridge-opencode-plugin](https://github.com/YuanG1944/message-bridge-opencode-plugin)），通过 `@opencode-ai/sdk` 与本地 OpenCode 交互，经 AK/SK 认证后与 AI-Gateway 建立长连接。服务端拆分为两个独立服务：**AI-Gateway**（连接管理与透传）和 **Skill Server**（持久化、推流、IM 消息发送），两者之间通过 WebSocket 长连接传递流式数据。

本次实现**仅考虑 OpenCode 本地模式**，个人私有会话。Claude Code / OpenClaw / 远程模式均不在本次范围内。

## 系统边界

### 我们构建的 4 个组件

| 组件 | 性质 | 运行位置 | 职责 |
|---|---|---|---|
| **PC Agent 插件** | Electron Main Process 模块 | Windows PC 客户端 | 桥接本地 OpenCode ↔ AI-Gateway |
| **AI-Gateway** | 独立 Spring Boot 服务 | 服务端独立部署 | 连接管理、AK/SK 认证、透传中继 |
| **Skill 服务端** | 独立 Spring Boot 服务 | 服务端独立部署 | 会话持久化、客户端推流、IM 消息发送 |
| **Skill 客户端** | 小程序 | IM 客户端小程序容器 | "/" 触发、Mini Bar、覆盖式对话 UI |

### 不在边界内（已有/外部）

- IM 聊天客户端（已有，我们往里集成 PCAgent 插件 + 注册 SKILL）
- IM 服务端 / 平台 API（已有，Skill 服务端调用它发 IM 消息）
- "/" 命令框架（已有，我们注册 SKILL 项并处理回调）
- OpenCode（第三方工具，PC Agent 通过 SDK 连接）
- 聊天平台开放平台 AK/SK 体系（已有，AI-Gateway 验证它）

## 架构设计

### 设计原则

1. **Skill 交互嵌入 IM**：通过已有 "/" 命令框架触发，Skill 小程序固定在 IM 聊天窗上方
2. **PCAgent 使用原生 SDK**：直接使用 `@opencode-ai/sdk` 处理 OpenCode 协议
3. **AI-Gateway 和 Skill Server 均为纯透传**：不解析、不转换 OpenCode 的输出内容，原样中继
4. **协议适配在 Skill 客户端**：Skill 小程序负责解析 OpenCode 事件格式、渲染富文本（Markdown/代码块/Tool Use 等）
5. **两个独立服务**：AI-Gateway 和 Skill Server 独立部署，通过 WebSocket 长连接传递流式数据
6. **AK/SK 认证**：PCAgent 连接 AI-Gateway 须通过聊天平台开放平台 AK/SK 认证
7. **本次仅 OpenCode 本地**：不考虑 Claude Code、OpenClaw、远程模式

### 组件职责详解

#### 1. PCAgent Plugin（Electron Main Process）

参考 `message-bridge-opencode-plugin` 架构，PCAgent 是集成在聊天客户端中的桥接模块：

| 职责 | 说明 |
|---|---|
| OpenCode 通信 | `@opencode-ai/sdk`：`api.event.subscribe()` 订阅事件流，`api.session.chat()` 发送消息 |
| Gateway 连接 | 维持与 AI-Gateway 的持久 WebSocket，AK/SK 认证，断线指数退避重连（1s→2s→4s，上限 30s）|
| 事件中继 | OpenCode SSE 事件 → AI-Gateway；AI-Gateway 指令 → OpenCode SDK 调用 |
| 会话管理 | `api.session.create()` / `api.session.list()` 管理 OpenCode 侧会话 |
| 健康检测 | 定期 `api.global.health()` 检测 OpenCode 可达性，上报状态 |
| 降级处理 | 主事件流退化时切换备用流（参考 message-bridge-opencode-plugin 降级机制）|

```typescript
// PCAgent 核心结构伪码
class PcAgentPlugin {
  private gateway: GatewayConnection;
  private opencode: OpenCode;

  async start(config: { ak: string; sk: string; gatewayUrl: string }) {
    // 1. AK/SK 认证 → 建立 Gateway WebSocket
    this.gateway = new GatewayConnection(config.gatewayUrl);
    await this.gateway.connect(AkSkAuth.sign(config.ak, config.sk));

    // 2. 初始化 OpenCode SDK
    this.opencode = new OpenCode({ baseURL: 'http://localhost:54321' });

    // 3. 订阅 OpenCode 事件流 → 透传给 AI-Gateway
    const stream = await this.opencode.event.subscribe();
    for await (const event of stream) {
      this.gateway.send({ type: 'tool_event', sessionId: event.sessionId, event });
    }

    // 4. 监听 AI-Gateway 指令 → 调用 OpenCode SDK
    this.gateway.onMessage(async (msg) => {
      if (msg.type === 'invoke' && msg.action === 'chat') {
        await this.opencode.session.chat(msg.payload.toolSessionId, {
          text: msg.payload.text
        });
      } else if (msg.type === 'invoke' && msg.action === 'create_session') {
        const session = await this.opencode.session.create(msg.payload);
        this.gateway.send({ type: 'session_created', session });
      }
    });
  }
}
```

#### 2. AI-Gateway（独立 Spring Boot 服务）

AI-Gateway 是连接管理层，**不解析 AI 工具输出内容**：

| 职责 | 说明 |
|---|---|
| PCAgent 连接管理 | WebSocket 端点，接收 PCAgent 连接，维护连接池 |
| AK/SK 认证 | 验证 PCAgent 携带的 AK/SK 签名 |
| Agent 注册 | 记录在线 PCAgent 的设备信息、工具版本、连接状态 |
| 下行中继 | Skill Server → AI-Gateway → PCAgent（用户消息、会话指令）|
| 上行中继 | PCAgent → AI-Gateway → Skill Server（OpenCode 流式输出）|
| Skill Server 连接 | 维持与 Skill Server 的 WebSocket 长连接，双向传递数据 |
| 心跳管理 | 接收 PCAgent 心跳，超时标记离线并通知 Skill Server |

#### 3. Skill Server（独立 Spring Boot 服务）

Skill Server 是业务逻辑层：

| 职责 | 说明 |
|---|---|
| SKILL 管理 | SKILL 定义查询（供 "/" 选择器使用）|
| 会话管理 | 创建/查询/关闭 Skill 会话 |
| 消息持久化 | 保存用户消息和 AI 响应到 MySQL |
| 客户端推流 | 通过 WebSocket 将 AI 流式响应实时推送给 Skill 小程序 |
| IM 消息发送 | 调用 IM 平台 API，将用户选中文本发送到 IM 聊天 |
| 历史恢复 | 提供消息历史查询接口，Skill 小程序打开时加载 |
| Gateway 通信 | 维持与 AI-Gateway 的 WebSocket 长连接，接收 AI 输出，下发用户指令 |

#### 4. Skill 客户端（小程序）— 协议适配层

Skill 客户端作为小程序运行在 IM 的小程序容器中，固定在 IM 聊天窗上方。**由于 AI-Gateway 和 Skill Server 都是透传 OpenCode 的原始数据，Skill 小程序承担协议兼容适配的职责**：

| 职责 | 说明 |
|---|---|
| **OpenCode 协议适配** | 解析 OpenCode 事件格式（message.updated / session.* / tool.* 等），转换为可渲染的 UI 状态 |
| **流式消息拼接** | 接收逐字 delta 事件，拼接为完整消息内容，处理中间状态和最终状态 |
| **富文本渲染** | Markdown 渲染、代码块语法高亮（shiki）、Tool Use 结果展示 |
| SKILL 触发 | 响应 IM "/" 框架回调，展示高亮 SKILL 行，接受用户问题输入 |
| Mini Bar | 1 行高状态条（处理中/完成/离线/展开按钮） |
| 对话视图 | 展开后的完整多轮对话界面 |
| 选文发送 IM | 选中 AI 回复文本 → 调用 Skill Server API → 发送到 IM 聊天 |
| 会话管理 | 会话列表、切换、历史加载 |
| **版本兼容** | 若 OpenCode 事件格式变更，仅需更新 Skill 小程序的解析逻辑，服务端无需改动 |

### 认证流程（AK/SK）

```
1. 管理员在聊天平台开放平台创建应用，获得 AK（Access Key）和 SK（Secret Key）
2. 用户在 Electron 客户端设置中配置 AK/SK
3. PCAgent 启动时：
   a. 用 SK 对 timestamp + nonce 签名，生成 signature
   b. 携带 AK + timestamp + nonce + signature 发起 WebSocket 握手
4. AI-Gateway 验证签名：
   a. 根据 AK 查询对应 SK
   b. 用同样算法计算签名，比对
   c. 检查 timestamp 防重放（5min 窗口）
5. 验证通过 → 建立持久 WebSocket 连接
```

### AI-Gateway ↔ Skill Server 通信协议

两个独立服务之间通过 **WebSocket 长连接**双向通信：

**连接建立**：AI-Gateway 启动时主动连接 Skill Server 的 WebSocket 端点（内部服务间通信，无需 AK/SK，使用内部 token 认证）。

```
# AI-Gateway → Skill Server（上行：AI 工具输出）
{"type":"tool_event","agentId":"...","sessionId":"...","event":{...}}
{"type":"tool_done","agentId":"...","sessionId":"...","usage":{...}}
{"type":"tool_error","agentId":"...","sessionId":"...","error":"..."}
{"type":"agent_online","agentId":"...","toolType":"OPENCODE","toolVersion":"1.2.15"}
{"type":"agent_offline","agentId":"..."}
{"type":"session_created","agentId":"...","toolSessionId":"..."}

# Skill Server → AI-Gateway（下行：用户指令）
{"type":"invoke","agentId":"...","sessionId":"...","action":"chat","payload":{"text":"...","toolSessionId":"..."}}
{"type":"invoke","agentId":"...","sessionId":"...","action":"create_session","payload":{}}
{"type":"invoke","agentId":"...","sessionId":"...","action":"close_session","payload":{"toolSessionId":"..."}}
```

### 协议兼容策略

**核心原则**：协议适配全部在 Skill 小程序完成，AI-Gateway 和 Skill Server 对 OpenCode 数据完全透传。

**本次实现（OpenCode 本地）**：
- PCAgent 通过 `@opencode-ai/sdk` 订阅 OpenCode SSE 事件流，原样透传给 AI-Gateway
- AI-Gateway → Skill Server → Skill 小程序全链路透传
- Skill 小程序内的 `protocol/` 模块负责解析 OpenCode 事件格式：
  - `OpenCodeEventParser.ts`：识别事件类型（`message.updated` / `session.completed` / `tool.*` 等）
  - `StreamAssembler.ts`：将多个 delta 事件拼接为完整消息
  - `ToolUseRenderer.ts`：将 Tool Use 结果转为可渲染结构

**版本兼容**：当 OpenCode 更新事件格式时，仅需更新 Skill 小程序的 `protocol/` 模块，服务端（AI-Gateway / Skill Server）和 PCAgent 无需改动。

**后续多工具扩展**（不在本次范围）：
- 新增 Claude Code / OpenClaw 时，Skill 小程序新增对应 `protocol/ClaudeCodeParser.ts` / `protocol/OpenClawParser.ts`
- PCAgent 新增对应 Bridge 模块（SubprocessBridge / WebSocketBridge）

## 数据模型（MySQL）

### Skill Server 数据库

```sql

-- Skill 会话
CREATE TABLE skill_session (
  id                  BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id             BIGINT NOT NULL,
  skill_definition_id BIGINT NOT NULL,
  agent_id            BIGINT,                       -- 关联的 PCAgent（AI-Gateway 侧）
  tool_session_id     VARCHAR(128),                  -- AI 工具侧的 session id
  title               VARCHAR(200),
  status              ENUM('ACTIVE','IDLE','CLOSED') DEFAULT 'ACTIVE',
  im_chat_id          VARCHAR(128),                  -- 关联的 IM 聊天 ID（用于发送 IM 消息）
  created_at          DATETIME NOT NULL,
  last_active_at      DATETIME NOT NULL,
  INDEX idx_user_active (user_id, status),
  INDEX idx_agent (agent_id)
);

-- Skill 消息
CREATE TABLE skill_message (
  id           BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id   BIGINT NOT NULL,
  seq          INT NOT NULL,                         -- 消息序号，防乱序
  role         ENUM('USER','ASSISTANT','SYSTEM','TOOL') NOT NULL,
  content      MEDIUMTEXT NOT NULL,
  content_type ENUM('MARKDOWN','CODE','PLAIN') DEFAULT 'MARKDOWN',
  created_at   DATETIME NOT NULL,
  meta         JSON,                                  -- token 用量、tool_use 等
  INDEX idx_session_seq (session_id, seq)
);
```

// 跨实例路由
public void routeToAgent(Long agentId, GatewayMessage msg) {
    String target = redis.get("gw:agent:conn:" + agentId);
    if (target.equals(selfInstanceId)) {
        localAgentPool.send(agentId, msg);
    } else {
        redis.publish("gw:relay:" + target, msg);
    }
}

## 风险与应对

| 风险 | 概率 | 应对 |
|---|---|---|
| OpenCode SDK API 变更 | 中 | `@opencode-ai/sdk` 版本锁定；PCAgent 按 semver 升级 |
| PCAgent WebSocket 断连 | 高 | 指数退避重连（1s→2s→4s，上限 30s）；AI-Gateway 心跳超时推 `agent_offline` |
| AI-Gateway ↔ Skill Server 连接断开 | 中 | Skill Server 端指数退避重连；断开期间缓冲到 Redis |
| Skill 小程序性能（大量消息）| 中 | 虚拟滚动（react-window）；消息分页加载（每次 50 条）|
| IM 消息发送权限 | 低 | AK/SK 权限校验；仅允许发送到用户自己的聊天 |
| OpenCode 本地未安装 | 高 | PCAgent 启动时检测可达性；不可达时 UI 显示安装引导 |
| 多 PCAgent 实例冲突 | 中 | 同一 AK + 同一工具类型只允许一个活跃连接；新连接踢掉旧连接 |
| 两个独立服务部署协调 | 中 | API 版本化；内部通信协议向后兼容；灰度发布策略 |
