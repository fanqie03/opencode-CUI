# opencode-CUI 项目架构分析总结

> **版本**：V1.0 | **日期**：2026-03-04 | **项目定位**：分布式事件驱动架构的 IM-AI 集成方案

---

## 1. 项目概述

### 1.1 项目名称与定位

**项目名称**：opencode-CUI（OpenCode Conversational User Interface）

**项目定位**：分布式事件驱动架构的 IM-AI 集成方案，通过透明中继模式实现 IM 客户端与本地 OpenCode 的无缝交互。

**核心价值**：
- 开发者无需离开 IM 即可与本地 OpenCode 进行多轮 AI 编码对话
- 通过 "/" 命令框架在 IM 中触发 Skill 交互
- 支持多用户、多设备、多会话的并发管理
- 实现高可用、可扩展的分布式架构

### 1.2 系统边界

**构建的 4 个核心组件**：

| 组件 | 性质 | 运行位置 | 职责 |
|------|------|---------|------|
| PC Agent 插件 | Electron 模块 | Windows PC 客户端 | 桥接本地 OpenCode ↔ AI-Gateway |
| AI-Gateway | Spring Boot 服务 | 服务端独立部署 | 连接管理、认证、事件中继 |
| Skill Server | Spring Boot 服务 | 服务端独立部署 | 会话管理、消息持久化、推流 |
| Skill 小程序 | React 应用 | IM 客户端容器 | 协议适配、富文本渲染、用户交互 |

**外部依赖**（不构建）：
- IM 聊天客户端与服务端
- "/" 命令框架
- OpenCode（第三方 IDE 工具）
- 聊天平台 AK/SK 体系

## 2. 架构亮点

### 2.1 透明中继模式

**核心设计**：服务端不解析 AI 工具输出，保持协议独立性。

```
OpenCode 原始事件
    ↓ (JSON 透传)
PCAgent event.subscribe()
    ↓ (WebSocket 透传)
AI-Gateway EventRelayService
    ↓ (WebSocket 透传)
Skill Server GatewayWSHandler
    ↓ (WebSocket 推流)
Skill 小程序 protocol/ 解析
    ↓ (类型化事件)
UI 渲染
```

**优势**：
- 服务端无需关心 OpenCode 版本更新
- 协议适配集中在客户端，易于维护
- 支持未来多工具扩展（VSCode、JetBrains 等）
- 低耦合、高内聚的架构设计

### 2.2 双服务架构职责分离

**AI-Gateway（端口 8081）**：
- 连接管理：PCAgent 注册、心跳、上下线
- 认证层：AK/SK 签名验证
- 中继层：双向消息透传（不解析）
- 无状态设计：支持水平扩展

**Skill Server（端口 8082）**：
- 会话管理：创建、查询、关闭
- 消息持久化：存储原始 OpenCode 事件
- 推流服务：WebSocket 推送给小程序
- IM 集成：调用 IM 平台 API 发送消息

**分离的好处**：
- Gateway 专注 I/O 密集的连接管理
- Skill Server 专注业务逻辑和数据持久化
- 独立扩展：Gateway 可处理更多连接，Skill Server 可处理更多业务

### 2.3 Redis Pub/Sub 协调

**多实例无状态设计**：

```
Agent 连接注册：
  Key:  gw:agent:conn:{agentId}
  Val:  {"instanceId":"gw-node-1","userId":123,"toolType":"OPENCODE"}
  TTL:  90s（心跳每 30s 刷新）

跨实例消息中继：
  Channel: gw:relay:{targetInstanceId}
  Message: GatewayMessage JSON
```

**路由逻辑**：
```
if (agentOnLocalInstance) {
    localAgentPool.send(agentId, msg);
} else {
    redis.publish("gw:relay:" + targetInstance, msg);
}
```

**优势**：
- 无状态：Gateway/Skill Server 实例可随意增删
- 自动发现：新实例自动订阅相关通道
- 高可用：Redis 故障时消息丢失可接受（有 WebSocket 直连备份）

### 2.4 客户端协议适配

**设计原则**：解析逻辑集中在 Skill 小程序 `protocol/` 模块。

**支持的 OpenCode 事件类型**：
- `message.updated` — 消息内容更新（含 delta）
- `session.completed` — 会话完成
- `session.created` — 会话创建
- `tool.start` — 工具调用开始
- `tool.result` — 工具调用结果
- `tool.error` — 工具调用错误

**版本兼容策略**：
- OpenCode 更新事件格式 → 仅需更新 Skill 小程序 `protocol/` 模块
- AI-Gateway、Skill Server、PCAgent 无需改动

## 4. 核心组件

### 4.1 AI-Gateway（端口 8081）

**职责**：连接管理、AK/SK 认证、事件透传、多实例协调

**关键类**：
- `AgentWebSocketHandler` — PCAgent 连接生命周期管理
- `AkSkAuthService` — HMAC-SHA256 签名验证
- `EventRelayService` — 双向消息透传
- `RedisMessageBroker` — Redis Pub/Sub 协调

**数据库表**：
```sql
CREATE TABLE agent_connection (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  ak_id VARCHAR(255) NOT NULL,
  device_name VARCHAR(255),
  os VARCHAR(50),
  tool_type VARCHAR(50),
  tool_version VARCHAR(50),
  status VARCHAR(20),
  last_seen_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 4.2 Skill Server（端口 8082）

**职责**：会话管理、消息持久化、客户端推流、IM 集成

**关键类**：
- `GatewayRelayService` — 接收 Gateway 消息
- `SkillStreamHandler` — WebSocket 推流给小程序
- `SkillSessionService` — 会话管理
- `SkillMessageService` — 消息持久化
- `ImMessageService` — IM 平台集成

**数据库表**：
```sql
CREATE TABLE skill_session (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  skill_definition_id BIGINT NOT NULL,
  agent_id BIGINT,
  title VARCHAR(255),
  im_chat_id VARCHAR(255),
  tool_session_id VARCHAR(255),
  status VARCHAR(20),
  last_active_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE skill_message (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  session_id BIGINT NOT NULL,
  message_type VARCHAR(20),
  content LONGTEXT,
  metadata JSON,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4.3 PC Agent Plugin

**职责**：桥接本地 OpenCode 与远端 Gateway

**关键模块**：
- `OpenCodeBridge` — @opencode-ai/sdk 封装
- `EventRelay` — OpenCode 事件 ↔ Gateway 双向中继
- `AkSkAuth` — AK/SK 签名认证
- `GatewayConnection` — WebSocket 长连接管理
- `HealthChecker` — OpenCode 可达性检测

**断线重连策略**：指数退避（1s → 2s → 4s → 8s → 16s → 30s）

### 4.4 Skill 小程序

**职责**：协议适配、富文本渲染、用户交互

**关键模块**：
- `protocol/OpenCodeEventParser.ts` — 解析 OpenCode 事件
- `protocol/StreamAssembler.ts` — 流式消息拼接
- `protocol/ToolUseRenderer.ts` — Tool Use 结果可视化

- `hooks/useSkillStream.ts` — WebSocket 推流订阅

## 5. 关键设计决策

### 5.1 为什么选择透传而非服务端解析

| 方案 | 优点 | 缺点 |
|------|------|------|
| **透传** | 协议独立、版本兼容、低耦合 | 无法在 Gateway 层做统一处理 |
| 解析 | 统一处理、易于监控 | 协议绑定、版本维护复杂 |

**选择理由**：
- OpenCode 事件格式复杂且可能变化
- 多工具支持需要协议独立性
- Gateway 只需负责路由，不需理解业务逻辑

### 5.2 为什么需要两个独立服务

| 方案 | 优点 | 缺点 |
|------|------|------|
| **双服务** | 职责清晰、独立扩展 | 部署复杂、通信开销 |
| 单服务 | 简单、部署少 | 职责混乱、扩展困难 |

**选择理由**：
- Gateway：连接管理、认证、路由（I/O 密集）
- Skill Server：会话管理、消息存储、推流（计算密集）
- 分离后可独立扩展，Gateway 可处理更多连接

### 5.3 为什么使用 WebSocket 长连接

| 方案 | 优点 | 缺点 |
|------|------|------|
| **WebSocket** | 低延迟、双向通信、连接复用 | 连接管理复杂 |
| HTTP Polling | 简单、无状态 | 高延迟、高开销 |
| gRPC | 高性能、类型安全 | 客户端支持差 |

**选择理由**：
- 实时推流需要低延迟
- IDE 工具原生支持 WebSocket
- 连接复用减少开销

### 5.4 为什么使用 Redis Pub/Sub

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Redis Pub/Sub** | 简单、低延迟、自动发现 | 消息不持久化 |
| RabbitMQ | 持久化、可靠性高 | 复杂、开销大 |
| Kafka | 高吞吐、持久化 | 复杂、延迟高 |

**选择理由**：
- 消息不需持久化（有 WebSocket 直连和数据库备份）
- 低延迟要求
- 简化部署和运维

## 6. 数据流概览

### 6.1 上行流：用户消息 → OpenCode

```
用户在 Skill 小程序输入消息
    ↓
POST /api/skill/sessions/{id}/messages
    ↓
Skill Server 持久化消息
    ↓
Skill Server 通过 WebSocket 发送到 AI-Gateway
    ↓
AI-Gateway 通过 WebSocket 发送到 PCAgent
    ↓
PCAgent 调用 opencode.session.chat()
    ↓
OpenCode 处理请求
```

### 6.2 下行流：OpenCode 响应 → 用户

```
OpenCode 生成 SSE 事件流
    ↓
PCAgent event.subscribe() 接收事件
    ↓
PCAgent 通过 WebSocket 透传到 AI-Gateway
    ↓
AI-Gateway 通过 WebSocket 透传到 Skill Server
    ↓
Skill Server 持久化消息到 MySQL
    ↓
Skill Server 通过 WebSocket 推流到 Skill 小程序
    ↓
Skill 小程序 protocol/ 模块解析事件
    ↓
UI 渲染消息
```

### 6.3 跨实例路由机制

**场景**：多个 Gateway 实例，PCAgent 连接到实例 A，Skill Server 实例 B 需要发送消息

```
Skill Server B 需要发送 invoke 到 agent:456
    ↓
GatewayRelayService.sendInvokeToGateway()
  ↓
Redis publish agent:456 (所有 Gateway 实例订阅)
    ↓
Gateway A 的 EventRelayService 接收
    ↓
检查本地是否有 agentId=456 的连接
    ↓
有：直接发送到 WebSocket
无：消息丢弃（该实例无此 agent）
```

### 6.4 选文发送到 IM 流

```
用户选中 AI 回复文本
    ↓
浮现"发送到聊天"按钮
  ↓
用户点击按钮
    ↓
Skill 小程序 POST /api/skill/sessions/{id}/send-to-im
    ↓
Skill Server 调用 IM 平台 API
    ↓
IM 聊天窗显示消息
```

## 7. 优化建议

### 7.1 添加 Redis 故障的熔断器

**问题**：Redis 不可用时，多实例协调失败

**建议**：
- 实现 Redis 连接池的熔断器模式
- 故障时自动降级为本地路由（单实例模式）
- 定期检测 Redis 恢复，自动恢复协调

**实现方式**：
```java
@CircuitBreaker(name = "redis", fallbackMethod = "localRouteFallback")
public void publishToRedis(String channel, String message) {
    redis.publish(channel, message);
}

private void localRouteFallback(String channel, String message) {
    // 降级为本地路由
    localAgentPool.send(extractAgentId(channel), message);
}
```

### 7.2 实现消息恢复协议

**问题**：网络波动导致消息丢失

**建议**：
- 为每条消息添加序列号（seq）
- 接收端检测序列号间隙
- 间隙时请求消息恢复

**实现方式**：
```
小间隙（1-5）：记录警告，继续处理
中间隙（6-20）：请求恢复
大间隙（>20）：触发重连
```

### 7.3 增加透明中继的可观测性

**问题**：透传模式下难以监控消息内容

**建议**：
- 添加消息计数器（按类型统计）
- 记录消息大小分布
- 添加链路追踪（Trace ID）
- 监控消息延迟

**关键指标**：
- 消息吞吐量（msg/s）
- 消息延迟（p50/p95/p99）
- 错误率（按类型）
- 连接数（按状态）

### 7.4 优化数据库查询性能

**问题**：消息表可能快速增长

**建议**：
- 添加分区策略（按 session_id 或时间）
- 实现消息归档（定期移动旧消息到冷存储）
- 添加查询缓存（Redis）
- 优化索引策略

**索引优化**：
```sql
CREATE INDEX idx_session_seq ON skill_message(session_id, seq);
CREATE INDEX idx_user_active ON skill_session(user_id, status);
CREATE INDEX idx_agent ON skill_session(agent_id);
```

## 9. 部署架构

### 9.1 开发环境

```
localhost:3306 (MariaDB)
  ├─ ai_gateway (Gateway 数据库)
  └─ skill_db (Skill Server 数据库)

localhost:6379 (Redis)

localhost:8081 (AI-Gateway)
localhost:8082 (Skill-Server)
localhost:3000 (Skill Miniapp)
```

### 9.2 生产环境（高可用）

```
Load Balancer (L7, 支持 WS upgrade)
    │
    ├─ [GW Node 1]
    ├─ [GW Node 2]
    └─ [GW Node 3]
        │
     └─ Redis Cluster
            │
            ├─ [SS Node 1]
            ├─ [SS Node 2]
            └─ [SS Node 3]
           │
                └─ MariaDB Cluster
```

**配置要点**：
- Load Balancer 支持 WebSocket upgrade
- Redis 使用 Cluster 模式
- MariaDB 使用主从复制或 Galera Cluster
- 每个服务配置独立的数据库连接池

### 9.3 启动顺序

1. MariaDB
2. Redis
3. AI-Gateway
4. Skill Server
5. Skill 小程序

## 11. 总结

opencode-CUI 通过分布式事件驱动架构实现了高效的 IDE-AI 集成。核心设计包括：

1. **透明中继**：Gateway 不解析消息，保持协议独立
2. **Redis 协调**：实现无状态的多实例扩展
3. **会话持久化**：所有消息存储，支持离线恢复
4. **实时推流**：WebSocket 长连接提供低延迟体验
5. **安全认证**：AK/SK 签名 + Nonce 防重放

该架构支持水平扩展、多工具接入、版本兼容，是一个生产级的分布式系统设计。
