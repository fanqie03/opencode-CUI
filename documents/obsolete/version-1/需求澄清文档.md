## 一、项目定位调整

### 原始定位
分布式事件驱动架构的 IM-AI 集成方案，通过透明中继模式连接 IM 客户端与本地 OpenCode。

### 调整后定位
**先开发本地联调 Demo**，验证核心技术链路的可行性，暂不依赖 IM 平台集成。

**核心变化**：
- ✅ 使用 Web UI 模拟 IM 客户端和 Skill 客户端的交互
- ✅ Web UI 通过 WebSocket 与 Skill Server 建立长连接
- ✅ PCAgent 已存在于项目代码中，需要验证和完善
- ✅ 实现完整的本地联调链路：OpenCode → PCAgent → AI-Gateway → Skill Server → Web UI

## 二、需求澄清结果

### 2.1 Critical 级别（必须在开发前明确）

#### IM 平台集成
**决策**: 先通过 Web UI 模拟 IM 客户端和 Skill 客户端的交互，暂不依赖 IM 平台。

**理由**:
- 降低外部依赖，加快 demo 开发速度
- 先验证核心技术链路的可行性
- 后续可以基于 demo 与 IM 平台团队进行技术对接

#### PCAgent 插件
**决策**: PCAgent 已经开发完成，位于项目代码中。

**目标**: 实现一个完整的本地联调 demo，不依赖 IM 客户端，直接实现 OpenCode → PCAgent → AI-Gateway → Skill Server → Web UI 的端到端链路。

#### OpenCode 部署要求
**决策**: 使用默认配置即可，无特殊要求。

**说明**:
- 用户本地安装标准版 OpenCode
- 使用默认的 `localhost:54321` 端口
- 无需特殊启动参数或配置文件

#### AK/SK 管理
**决策**: 需要开发管理后台，支持用户自助申请和管理。

**功能范围**:
- 用户注册和登录
- 创建、查看、删除 AK/SK
- 基础的权限管理
- 后续扩展：轮换、撤销、使用统计

---

### 2.2 High 级别（Phase 1 开发期间必须明确）

#### Demo 功能范围
**决策**: 包含以下核心功能：

1. **完整的 OpenCode 事件透传和渲染**
   - 支持所有事件类型：message.updated、tool.start、tool.result、tool.error、session.*
   - 验证协议适配层的正确性

2. **会话历史的持久化和恢复**
   - 关闭页面后重新打开能恢复完整对话
   - 包括代码块、Tool Use 结果等所有内容

3. **多实例协调（Redis Pub/Sub）**
   - 验证多个 Gateway/SkillServer 实例的消息路由
   - 测试实例故障时的自动切换

4. **AK/SK 管理后台的基础功能**
   - 用户注册/登录
   - 创建、查看、删除 AK/SK
   - 简单的权限验证

#### 消息大小限制
**决策**: Demo 阶段不做限制。

**说明**:
- 允许超大响应（如长代码、详细日志）
- 生产环境再根据实际情况设置限制（建议 10MB）

#### 并发会话限制
**决策**: 需要限制，单用户最多 10 个活跃会话。

**理由**:
- 防止资源滥用
- 提供更好的用户体验（避免会话过多导致混乱）
#### 消息保留期限
**决策**: 永久保留。

**说明**:
- 所有消息持久化到数据库
- 用户可以手动删除会话和消息
- 后续可以根据存储成本考虑归档策略

### 2.3 用户体验细节

#### WebSocket 连接断开处理
**决策**: 自动重连（推荐）。

**策略**:
- 连接断开后自动重试，最多 3 次
- 使用指数退避（1s / 2s / 4s）
- 重试失败后提示用户手动刷新

#### OpenCode 离线处理
**决策**: 允许离线消息排队。

**行为**:
- 显示 "OpenCode 离线" 状态
- 用户可以继续发送消息，消息进入队列
- OpenCode 上线后自动发送排队的消息
- 超过 5 分钟未上线则提示用户检查 OpenCode 状态

#### 消息渲染功能
**决策**: 支持以下功能：

1. **Markdown 渲染**
   - 标题、列表、引用、链接等
   - 使用 `react-markdown` 库

2. **代码语法高亮**
   - 支持主流编程语言
   - 使用 `shiki` 或 `prism` 库

3. **Tool Use 可视化**
   - 显示工具调用的输入和输出
   - 区分不同的工具类型（文件读写、终端命令等）
   - 支持折叠/展开

4. **流式渲染**
   - 逐字显示 AI 回复
   - 实时更新代码块和 Tool Use 结果

#### 多设备登录
**决策**: 单设备登录（推荐）。

**行为**:
- 新设备登录时踢掉旧设备
- 旧设备显示 "您的账号在其他设备上登录，当前连接已断开"
- 后续可以扩展为多设备同时在线（需要消息同步机制）

### 2.4 技术选型

#### 部署方式
**决策**: 支持本地和服务端部署。

**本地部署**:
- 所有服务在本地启动
- 使用 Docker Compose 或启动脚本管理
- 适合开发和演示

**服务端部署**:
- 部署到开发/测试服务器
- 支持远程访问
- 适合团队协作和测试

#### 数据库
**决策**: MySQL 5.7。

**说明**:
- 与生产环境保持一致
- 使用 HikariCP 连接池
- 支持主从复制（生产环境）

#### 前端技术栈
**决策**: React + TypeScript + WebSocket。

**技术栈**:
- React 18
- TypeScript 5
- WebSocket 原生 API 或 `socket.io-client`
- `react-markdown` + `shiki` 用于内容渲染
- Tailwind CSS 或 Ant Design 用于 UI

**说明**:
- 项目中已有 `skill-miniapp` 目录，可以基于现有代码扩展
- 如果现有代码不满足需求，可以重新开发

#### 开发优先级
**决策**: 核心链路优先（推荐）。

**实施步骤**:
1. **Phase 1**: 实现核心链路（OpenCode → PCAgent → Gateway → SkillServer → WebUI）
   - 验证 PCAgent 的事件订阅和转发
   - 验证 Gateway 的 WebSocket 连接管理
   - 验证 SkillServer 的消息路由和持久化
   - 实现 WebUI 的基础消息渲染

2. **Phase 2**: 完善功能
   - 实现流式渲染
   - 实现 Tool Use 可视化
   - 实现会话历史恢复
   - 实现多实例协调

3. **Phase 3**: 开发 AK/SK 管理后台
   - 用户注册/登录
   - AK/SK 的 CRUD 操作
   - 权限验证

4. **Phase 4**: 优化和测试
   - 性能优化
   - 错误处理
   - 端到端测试

## 三、架构调整

### 3.1 原始架构
```
IM 客户端 (PCAgent 插件)
  ↓ WebSocket
AI-Gateway (连接管理)
    ↓ Redis Pub/Sub
Skill Server (业务逻辑)
    ↓ WebSocket
IM 客户端 (Skill 小程序)
```

### 3.2 调整后架构（Demo）
```
OpenCode (本地实例)
    ↓ Event Subscribe
PCAgent (独立进程)
    ↓ WebSocket
AI-Gateway (连接管理)
    ↓ Redis Pub/Sub
Skill Server (业务逻辑)
    ↓ WebSocket
Web UI (React 应用)
```

### 3.3 关键变化
1. **PCAgent 独立运行**: 不再作为 IM 客户端插件，而是独立进程
2. **Web UI 替代 IM 小程序**: 使用 React 开发的 Web 应用，模拟 IM 客户端的交互
3. **保留多实例协调**: 仍然使用 Redis Pub/Sub 实现多实例协调，验证架构的可扩展性

## 六、风险和缓解措施

### 6.1 技术风险

#### 风险 1: PCAgent 的稳定性
**描述**: PCAgent 订阅 OpenCode 事件可能存在内存泄漏、连接断开、事件丢失等问题。

**缓解措施**:
- 在测试环境中运行 PCAgent 连续 24 小时，监控内存和 CPU 使用率
- 实现健康检查和自动重启机制
- 记录详细的错误日志，便于排查问题

#### 风险 2: Redis Pub/Sub 的消息丢失
**描述**: Redis 故障时，正在传输的消息会丢失，用户可能看到消息乱序或缺失。

**缓解措施**:
- 实现消息序列号检测和恢复协议
- 考虑使用 Redis Stream 替代 Pub/Sub（提供持久化）
- 降级为本地路由（单实例模式）

#### 风险 3: WebSocket 长连接的稳定性
**描述**: WebSocket 连接可能因为网络抖动、防火墙、代理等原因断开。

**缓解措施**:
- 实现心跳保活机制（30 秒间隔）
- 实现自动重连机制（最多 3 次）
- 使用 WSS（WebSocket over TLS）提高安全性

### 6.2 进度风险
#### 风险 1: 前端开发时间不足
**描述**: React + TypeScript + WebSocket 的开发可能比预期复杂。

**缓解措施**:
- 优先实现核心功能，UI 美化延后
- 使用成熟的 UI 组件库（Ant Design）
- 参考现有的 `skill-miniapp` 代码

#### 风险 2: 多实例协调的复杂度
**描述**: Redis Pub/Sub 的消息路由和序列号检测可能存在边界情况。

**缓解措施**:
- 先实现单实例模式，验证核心链路
- 再实现多实例协调，逐步测试
- 编写详细的集成测试用例

## 八、附录

### 8.1 关键决策记录

| 决策项 | 原方案 | 调整后方案 | 理由 |
|--------|---------|------------|------|
| IM 集成 | 依赖 IM 平台的 "/" 框架和小程序容器 | 使用 Web UI 模拟 | 降低外部依赖，加快开发速度 |
| PCAgent | 需要开发 | 已存在于项目代码中 | 可以直接使用和验证 |
| 数据库 | MariaDB | MySQL 5.7 | 用户指定 |
| 消息大小限制 | 10MB | 不限制（demo 阶段） | 简化开发，后续再优化 |
| 部署方式 | 本地部署 | 支持本地和服务端部署 | 提高灵活性 |

### 8.2 术语表

| 术语 | 说明 |
|------|------|
| PCAgent | 运行在用户本地的代理程序，负责订阅 OpenCode 事件并转发到 AI-Gateway |
| AI-Gateway | WebSocket 连接管理服务，负责 AK/SK 验证和消息路由 |
| Skill Server | 业务逻辑服务，负责消息持久化、会话管理、与 AI 工具交互 |
| Web UI | React 开发的 Web 应用，模拟 IM 客户端的交互界面 |
| 透明中继 | 服务端不解析 OpenCode 事件，协议适配在客户端完成 |
| 多实例协调 | 使用 Redis Pub/Sub 实现多个 Gateway/SkillServer 实例的消息路由 |
